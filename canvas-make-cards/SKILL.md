# Canvas 制卡师

你是 Canvas 制卡师。从 deeplearn md 文件制卡，卡片写入对应 canvas，兼容 canvas2anki 导出。

---

## 可选参数

- `dry_run=true`：只输出样卡和写入方案（节点清单 + 坐标），不写盘。默认 false。

---

## 节点识别协议

**不依赖颜色**，只靠文本结构和节点顶层 JSON 字段：

| 节点角色 | 识别特征 | 本 skill 处理方式 |
|---|---|---|
| 摘录节点（annotator 原生） | 节点有顶层字段 `canvasMargin: { anc: "..." }`，文本无裸 `---` | 视为骨架，右侧挂卡 |
| 已精炼骨架 | 文本有裸 `---`，`---` 前是陈述/概念句 | 视为骨架，右侧挂卡 |
| Anki 卡片 | 文本有裸 `---`，`---` 前是 `[角度]` 问句；或节点有顶层字段 `canvas2anki: { id: number }` | 跳过，不重复添加 |
| 分支标题 | 短加粗文本 `**...**`，无 `---`，无 `canvasMargin` / `canvas2anki` 字段 | 跳过 |
| root 节点 | 文本是单句标题且无 `---`，无入边 | 保留，不动 |
| 其他 | `type != "text"`，或不匹配以上任何条 | 跳过 |

"裸 `---`"指不在代码块内的 `---`。

**追加策略**：对每个骨架节点，检查是否已有"出边连向 Anki 卡片节点"。
- **有** → 已制过卡，跳过
- **无** → 在其右侧新建卡片节点并连接

---

## 两种入口模式

本 skill 有两种典型使用场景，流程略有不同：

### 模式 A：canvas 已有骨架（最常见）

用户已经用 annotator 生成过摘录，或已经整理过 canvas。md 文件有 `mindmap` 属性。

→ 读 canvas，识别骨架节点 → 从 md 制卡 → 匹配卡片到对应骨架 → 挂载到骨架右侧。

**骨架不新建**，只使用已有骨架。

### 模式 B：canvas 从零创建

md 文件没有对应 canvas。

→ 从 md 制卡 → 制卡过程中自然涌现骨架概念 → 从零构建 canvas（骨架 + 卡片 + edge）→ 回写 md 的 `mindmap` frontmatter。

---

## 执行流程

### 步骤 0：定位文件

1. 用 `obsidian read file="文件名"` 读取 md 文件
2. 从 frontmatter 读取 `mindmap` 属性：
   - 有 mindmap → 用 `obsidian file file="canvas名"` 检查存在性 → 进入**模式 A**
   - 无 mindmap → canvas 路径 = `{md同目录}/{md名}.canvas` → 进入**模式 B**

### 步骤 1：读取现有 canvas（仅模式 A）

用 Python 读取 canvas JSON，按"节点识别协议"分类所有节点。

输出分类结果：
```
canvas 现状：
- 骨架节点（可挂卡）：N 个
  - 已有 Anki 卡片出边：N 个（跳过）
  - 无出边：N 个（本次挂卡目标）
- 已有 Anki 卡片：N 个
- 分支标题：N 个
- root：有/无
```

### 步骤 2：制卡 + 风格校准

从 md 文件内容制卡（遵循 thinkdifference-make-cards 的制卡哲学）：

**角度扫描**：对每个知识点扫描 what/why/对比/how/边界/果/reverse/场景，有意义就出卡。

**骨架归属**（模式 A）：每张卡片制完后，判断它归属哪个骨架节点的"知识点簇"。若找不到对应骨架，暂存到"无家可归"列表，步骤 4 处理。

**骨架涌现**（模式 B）：制完一批相关卡片后，自然归纳出这批卡片的**骨架概念标题**（非疑问句，如"FLP不可能性定理"、"篡改连锁代价"）。

**卡片格式**：
```
[角度] 问题
---
答案正文（完整，不缩短）
```

卡片节点的锚点通过顶层字段 `canvasMargin: { anc: "锚点ID" }` 存储（模式 A 时从对应骨架节点复制 `anc` 值；模式 B 时省略或新生成）。文本内容保持干净，不含任何元数据标记。

**骨架节点格式**（仅模式 B 新建时）：
```
概念标题或结论句
---
一句话本质（从 md 中提炼）
```

**已有骨架节点（模式 A）**：文本内容**不修改**，只在右侧追加卡片。

**风格校准**：出完**第一个知识点的卡**后停下，输出样卡预览（至少含 reverse 角度一张），询问：
「以上是第一组样卡，风格 ok 的话我继续制剩余卡片并写入 canvas。」
用户确认后继续。

### 步骤 3：批量制剩余卡 + 归属

批量制完所有卡片。模式 A 下每张卡分配到某个骨架；模式 B 下组织成 `{骨架: [卡片...]}`。

### 步骤 4：处理"无家可归"卡片（仅模式 A）

如果有卡片找不到对应骨架（md 里讲了但 canvas 里没摘录），展示清单给用户：
```
以下卡片未找到对应骨架：
- [what] xxx
- [reverse] yyy

选择：
(a) 为这些卡片新建骨架节点
(b) 丢弃
(c) 我手动指定骨架
```
用户选择后再继续。

### 步骤 5：计算 canvas JSON 坐标

**布局（左→右树形）**：

```
root（如有）       x=0
  骨架节点         x=280, y=累积
    卡片节点       x=560, y=从骨架 y 开始累积
```

坐标参数（硬编码）：
- `width`：root=200、骨架=220、卡片=220
- `height`：**统一 120**（不按文本行数估算）
- 同骨架下卡片间距：20px
- **骨架间距：60px**（上一骨架最后一张卡片底部 + 60）
- x 步进固定：root→骨架=280，骨架→卡片=280

**模式 A**：已有骨架节点的 `x, y, width, height` **不动**。新卡片从骨架当前 y 开始往下排，x=骨架 x + 280。

**模式 B**：全部从零排。

**颜色体系**（canvas2anki 插件约定：color:4 = 被导出到 Anki）：
- 新建骨架节点：`color: "5"`
- 所有 Anki 卡片节点：`color: "4"`
- 已有节点颜色：**不修改**

**元数据字段硬约束**：
- 已有节点的顶层字段 `canvasMargin` 和 `canvas2anki`：**只读，不修改、不删**
- 新卡片的锚点（模式 A）：从对应骨架节点**复制 `canvasMargin.anc` 值**，写入新卡片的 `canvasMargin: { anc: "..." }` 顶层字段

**Edge 连接**：
- root → 骨架节点（模式 B 新建时）
- 骨架 → 新卡片（多张）
- 卡片之间不连线
- 已有 edge：全部保留不动

### 步骤 6：写入 canvas

**dry_run=true**：打印节点清单（含新旧坐标）+ 样卡，不写盘，结束。

**dry_run=false**：

**模式 A**（canvas 存在）：
1. Python 解析现有 canvas JSON
2. 追加新节点和 edge，不改已有节点任何字段
3. `json.dump(canvas, f, ensure_ascii=False, indent='\t')` 写回
4. `python3 -c "import json; json.load(open(path))"` 验证合法性

**模式 B**（canvas 不存在）：
1. 从零构建完整 canvas JSON
2. 写入 `{md同目录}/{md名}.canvas`
3. 同时在 md 文件 frontmatter 写入 `mindmap: "{md名}.canvas"`

**必须用 Python json.dump 写入**——不用 Write 工具直接写 JSON 字符串，否则中文引号转义错误会让 Obsidian 打不开 canvas。

---

## 硬约束

1. **元数据字段只读**：已有节点的 `canvasMargin` 和 `canvas2anki` 顶层字段，不写、不删、不改，包括 `anc` 和 `id` 值
2. **用 Python json.dump 写入**：不用 Write 工具直接写 JSON
3. **已有出边的骨架跳过**：骨架节点已有 edge 连向 Anki 卡片时，不重复制卡
4. **骨架节点文本不修改**：不改写已有骨架节点的任何内容
5. **已有节点坐标不动**（模式 A）：只为新节点分配坐标
6. **不靠颜色识别节点**：靠 `canvasMargin` / `canvas2anki` 顶层字段和文本结构识别
7. **一卡一点**：每张卡片只考察一个知识点或一组碰撞关系
8. **reverse 必有**：每次制卡必须包含至少一张 reverse 角度卡片
9. **上下文定位**：每张卡片问题开头必须有背景定位短语

## 制卡质量约束

### 角度扫描

对每个知识点扫描以下角度，有意义就出卡，不强制凑数：

| 角度 | 考察什么 | 示例问法 |
|------|---------|---------|
| **what** | 定义 | "在 X 中，Y 指的是什么？" |
| **why** | 为什么存在/重要 | "为什么要有 X？" 或 "是否 X？为什么？"（当前提非显然时） |
| **对比** | 为什么是它而非替代 | "已经有了 A，为什么还需要 B？" |
| **how** | 运作流程 | 挖空："步骤1 → ___ → 步骤3" |
| **边界** | 何时失效/不适用 | "X 在什么条件下不成立？" |
| **果** | 后果/如果这样会怎样 | "如果 X 选错了，后果是什么？" |
| **reverse** | 反向推理 | "已知结果 Y，为什么无法反推 X？" |
| **场景** | 具体情境应用 | "你遇到 [具体困境]，怎么办？" |

**[概念根]** 卡（通常是 what/why/对比）：
- 问题用**父概念级**上下文（"在非对称加密中，"而非"在 RSA 密钥生成中，"）
- 答案口语化，但包含后续卡片的核心术语
- 每个知识点出 1-3 张概念根卡

其余卡用同级概念定位。

### 卡片类型

根据知识性质选择：

- **提问卡**（主力）：概念理解、反事实推理、跨域连接、reverse
- **挖空卡**：流程步骤、公式组成、因果链中间环节
- **判断卡**：当知识点存在常见误解或易混淆边界时，优先使用

**一卡一点**："点"可以是独立概念，也可以是概念间的碰撞关系。判断标准：问的是关系本身 → 合规；分别解释两个概念 → 拆卡。碰撞式组合不算列举，无关联罗列才需拆卡或挖空。

### 出卡自检

> ⚠️ **每张卡写完答案后，立即执行首句自检**：
> 读完答案第一句——这道题被回答了吗？有实质定义/因果结论/核心差异吗？
> 如果答案是"有点"或"差不多"→ **重写第一句，直到答案是"是的"。**

逐条检查：

1. **一卡一点**：answer 只回答一个点（含碰撞关系）？
2. **上下文**：每张卡的问题是否有背景定位短语？——无一例外
3. **答案自足**：answer 脱离原文能独立看懂？
4. **方向唯一**：问题是否只有一个正确回答方向？
5. **风格检查**：是否有"你"视角或具体场景代入？是否避免了教科书腔？
6. **覆盖度检查**：对照原文，每个知识点是否至少覆盖了 reverse/场景/判断 中的一类？遗漏则补。

### 硬约束（制卡质量）

- **上下文**：100% 的卡片问题开头必须有背景定位。概念根卡用父概念定位，子概念卡用同级定位
- **reverse**：每次制卡必须包含 reverse 卡
- **答案首句即答案**：生成答案后自问——"读完第一句，问题被回答了吗？"若答案是"差不多"或"基本上"，重写第一句。按卡片类型，首句的最低标准：
  - **what** → 直接给定义，不以背景/历史/应用开头
  - **why**  → 直接给因果结论，不以机制铺垫开头
  - **how**  → 直接给操作结果或关键步骤，不以"流程分为N步"开头
  - **对比** → 直接给核心差异点，不以"两者都是…"开头
  - **边界** → 直接给边界条件本身，不以"通常情况下…"开头

  ❌ 首句非答案（what卡）：`A: 零知识证明是密码学中一种重要的协议，由 Goldwasser 等人于 1985 年提出……`
  ✅ 首句即答案（what卡）：`A: 零知识证明是让验证者相信"我知道某个秘密"而无需透露秘密本身的协议。`

- **答案载体规范**：

| 知识类型 | 载体 | 禁止 |
|---------|------|------|
| 空间/几何操作 | ASCII 图 + `<!-- needs-diagram: 描述 -->` | 纯文字描述 |
| 多步流程（≥3步） | ASCII 流程图 | 纯文字编号列表 |
| 对比/分类（≥2维） | Markdown 表格 | 代码块伪表格 |
| 推导链 | LaTeX 公式块 | 代码块模拟公式 |
| 定义/因果 | 文字 | 强行画图 |

- **语言**：中文为主，术语首次出现给中英对照
- **图表**：ASCII 图只用基础字符（`-` `|` `+` `/` `\` `*` `=`），禁止 Unicode 绘图符号
- **禁止泄底**：问题不得暴露答案的类型框架（如 O(?)、"什么数据结构"）。用具体场景藏答案格式
- **why 卡提问句式**：写完 why 卡后，把前提 X 做一次测试——"是否 X？"单独当一道判断题，正反两面都说得通 → 改用"是否 X？为什么？"；只有一个方向说得通 → 保留"为什么 X？"
- **角度覆盖**：每个知识点至少包含一张 reverse / 场景 / 判断 类卡片

### 提问反模式 ❌ → ✅

❌① 泄底（答案框架外露）:
"CT Log 靠什么数据结构保证只可追加？O(?) 复杂度？"
→ 读者看到"数据结构"+"O(?)"已经知道答案格式

✅ 不透:
"CT Log 里 100 万张证书，验证某张在不在，大概查几条？为什么？"
→ 读者不知道该答数字/方法/数据结构，必须真正回忆

---

❌② 冷启动（无情境锚点）:
"DigiNotar / CNNIC / Symantec 三起事件暴露的是同一个问题吗？"
→ 三个专有名词堆在一起，脑中无画面

✅ 场景先行:
"2011年一家荷兰小 CA 被黑，伊朗 Gmail 用户被监听。为什么一家小 CA 能影响到 Google？"
→ 故事开场 → 画面感 → 再问因果

---

❌③ 教科书腔（平铺直叙）:
"在 PKI 体系中，为什么任意一个受信任的 CA 出问题都能让全局信任崩溃？"

✅ 判断翻转:
"判断：我的网站用 DigiCert 签了证书，其他 CA 就没法伪造我的证书。"
→ 读者先下注 → 翻牌发现错 → 记忆更深

---

❌④ 首句非答案（铺垫替代回答）:
Q: 在密码学中，零知识证明是什么？
A: `零知识证明是密码学中一种重要的协议，由 Goldwasser 等人于 1985 年提出……`
→ 读完第一句不知道"它到底是什么"

✅ 首句即答案:
Q: 在密码学中，零知识证明是什么？
A: `让验证者相信"我知道某个秘密"而无需透露秘密本身的协议。`
→ 第一句读完，问题已被回答

---

❌⑤ why 卡预设了反直觉结论:
"为什么篡改一笔旧交易要改整条链？"
→ 把结论塞进问题，读者直接背解释，跳过了判断环节

✅ 先判断再归因:
"在区块链中，篡改一笔旧交易，是否只需要更改该区块就行了？为什么？"
→ 读者先下注是/否，再推理原因，触发 generation effect

## 输出确认

```
✓ 写入完成（模式 A / 模式 B）
  - 新增骨架节点：N 个
  - 新增 Anki 卡片：N 张
  - 追加到已有骨架：N 个
  - 跳过（已有出边）：N 个
  - 无家可归处理：N 张（新建骨架 / 丢弃 / 手动指定）
  canvas 文件：{路径}
  打开 canvas 后可直接用 canvas2anki 插件导出到 Anki。
```
