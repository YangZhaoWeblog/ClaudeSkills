# 最少墨水的焊接工

把散落的知识点焊到用户已有的知识网络上。不重走发明之路（那是 learn-deep），只做一次精准的焊接——找到焊接点、焊上去、验证焊牢了。

## 执行流程

### 第 1 步：定位宿主

用 obsidian CLI 找到这个知识点应该挂在哪个已有概念上：

```
obsidian search query="{关键词}" path="2.工作稿"
obsidian search query="{关键词}" path="1.原子笔记"
obsidian backlinks file="{可能的宿主概念名}"
```

确定文件路径：
- **找到宿主 deep-learn** → 同目录，命名 `{宿主概念}-light-learn.md`
- **只找到宿主原子笔记（无 deep-learn）** → 写到 `2.工作稿/{推断领域}/{宿主概念}-light-learn.md`
- **找不到任何宿主** → 写到 `0.Inbox/`，命名 `{概念}-light-learn.md`，在文件顶部注明"此条目尚无宿主 deep-learn，已暂存 Inbox，待宿主建立后迁移"，并向用户说明

**宿主不对怎么办**：如果用户暗示的宿主与知识点实际所属的机制层不符（如"CORS 和 TLS 有关的那个"），纠正宿主，在文件顶部写一个宿主修正说明块：
```
> ⚠️ 宿主修正：{用户暗示宿主} 与 {本概念} 实际上 {一句话说清为什么不对}。
> 已将宿主改为 [[{正确宿主}]]。
```

**多候选宿主**：若找到多个候选宿主，选语义最强的（机制根基而非使用场景），并在输出中简短说明："宿主候选：X / Y / Z，选 X，因为……"，让用户可以纠正。

**锚点无直接宿主时**：借用最近邻已有概念作为锚点 wikilink，并在锚点后注明"（借用上位概念）"。

如果目标 light-learn 文件已存在 → 追加模式。追加时同步更新 frontmatter 的 `updated` 字段为今日日期。

### 第 2 步：写条目

#### 新建文件时

```markdown
---
tags:
  - light-learn
  - {领域}
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
host: "[[{宿主概念}]]"
---

# {宿主概念} · 轻量补丁

## 目录

- [[#{条目标题}]]

---

## {条目标题}

**锚点**：你已经知道 [[{宿主概念}]] 是{______一句话回忆}。

{核心内容——见条件写作指引}

**一句话嫁接**：{______与宿主的最小关键差异}

**场景**：{______具体使用场景，2-3 行}

**反直觉检验**：
> Q：{______一个戳破类比弱点的问题}
> A：{______简短回答，2-3 行}
```

#### 追加条目时

在已有文件末尾追加新条目（使用完整五块骨架，不依赖前文），同时更新顶部目录：在 `## 目录` 下追加 `- [[#{新条目标题}]]`。

### 条件写作指引

核心内容块根据知识性质自适应：

- **如果新知识与已知共享机制、只是应用场景不同** → 用对比：表格或并列说清"哪里一样、哪里不一样"，优先用 small multiple（并排结构让差异自己说话）

- **如果新知识是一个孤立事实** → 用陈述：1-2 行事实 + 一行"所以它重要是因为……"或"在地图上的位置是……"

- **如果新知识是已有概念的极端情况或失效边界** → 用极端提问：先问"如果{极端条件}会怎样？"再给回答，最后说明这对已有理解的修正

### 第 3 步：不确定标记

对不确定的事实，行内标注 `[待验证]`，不编造。

### 第 4 步：升级信号与收尾

写完后执行两项检查：

**检查一：单条目复杂度**
若刚写完的条目核心内容块 ≥ 15 行，输出：
`⚠️ 此条目内容量较大，可能值得升级为独立 learn-deep。说"学透 {条目名}"开始深学。`

**检查二：文件累计条目数**
- **≥ 3 条** → 在输出末尾提示："💡 此文件已有 N 个条目。如果某个条目值得深挖，说'学透 {条目名}'升级为独立 learn-deep。"

**收尾提示（每次必须输出）**：
`说"制卡"走 make-cards，把这些条目转为 Anki 卡片。`

## 金色示例

以下是一个完整的 light-learn 条目，展示预期输出质量和长度：

```markdown
## mTLS（双向认证）

**锚点**：你已经知道 [[TLS]] 握手中，服务器出示证书、客户端验证身份，然后建立加密通道。

**对比**：

| | TLS | mTLS |
|---|---|---|
| 谁验证谁 | 客户端 → 服务器（单向） | 客户端 ⇄ 服务器（双向） |
| 客户端证书 | 不需要 | 需要，客户端也持有证书和私钥 |
| 握手差异 | — | 服务器多发 CertificateRequest，客户端多回 Certificate + CertificateVerify |

**一句话嫁接**：mTLS = TLS + 反过来再做一遍（服务器也要验客户端身份）。

**场景**：微服务之间互调 API——没有浏览器、没有人类用户，两个服务怎么互相确认身份？Kubernetes 的 service mesh（如 Istio）默认用 mTLS 保护 pod 间通信，每个服务都持证上岗。

**反直觉检验**：
> Q：既然 mTLS 更安全，为什么公网网站不都用 mTLS？
> A：因为公网用户是不可控的——你无法给全世界每个浏览器用户签发客户端证书。mTLS 适用于"参与方已知且可控"的封闭环境（企业内网、微服务、IoT），不适用于开放互联网。
```

## 硬约束

1. **不修改宿主文件** — light-learn 只写自己的文件，不碰宿主 deep-learn 或原子笔记
2. **不产出原子笔记或 Anki 卡片** — 那是 extract-atoms / make-cards 的工作。写完后可提示"说'制卡'走 make-cards"
3. **每条目必须五块** — 锚点 / 核心内容 / 一句话嫁接 / 场景 / 反直觉检验，缺一不可
4. **追加时更新目录** — 文件顶部 `## 目录` 保持与条目同步
5. **锚点必须含 wikilink** — 焊接点必须可追溯到已有知识网络
6. **不确定就标 [待验证]** — 宁可标注不确定，不编造细节

## 全局约束

- **语言**：中文为主，术语首现给中英对照
- **风格**：data-ink ratio 趋近 1，每行都传递信息，消灭装饰
- **概念名**：用 `[[wikilink]]` 格式
- **公式**：LaTeX（`$...$` 行内）
- **图表**：纯 ASCII（只用 `-|+/\*=`）
