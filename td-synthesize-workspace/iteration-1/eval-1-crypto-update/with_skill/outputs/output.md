# [[公钥密码学]]方案汇总 MOC

公钥密码学回答两个根本问题：
1. **保密** — 如何在公开信道上，让两个从未见过面的人建立只有他们知道的秘密？
2. **认证** — 如何证明"这话确实是我说的，而且没被改过"？

### 密钥协商 & 加密

- [[DH密钥交换-deeplearn]] — 双方协商，从不传密钥本身，秘密由双方各自"混合"生成
- [[RSA加密-deeplearn]] — 单方加密：用对方公钥锁箱，只有对方私钥能开
- [[ECC加解密-deeplearn]] — 把 DH/RSA 的数学地基搬到椭圆曲线上，更短的密钥实现同等安全
  - [[EC-ElGamal]] — 裸方案：点加法藏消息，教学用
  - [[ECIES]] — 工程版：共享秘密 + KDF + AES

### 数字签名

- [[数字签名-deeplearn]] — 私钥签，公钥验，证明身份 + 防篡改 + 不可抵赖
  - [[ECDSA]] — 椭圆曲线 + 临时随机数 $k$，64 字节签名（比特币/以太坊标配）

# [新增] ECC 三件套：同源分化与互补

> 同一个 [[椭圆曲线离散对数问题]]，衍生出三个协议。它们不是三个独立工具，而是同一块数学地基上长出的三个器官，各管一件事，在 [[TLS]] 中拼成完整安全链。

## 同源分化图

```
               私钥 d, 公钥 Q=dG
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
       [[ECDH]]   [[ECDHE]]   [[ECDSA]]
       静态协商    临时协商     数字签名
          │           │           │
          │     +临时密钥对       +随机数k
          │     +用完即毁        +Hash-then-Sign
          │           │           │
          ▼           ▼           ▼
       共享秘密    共享秘密     签名(r,s)
       (长期)    (一次性)     (可公开验证)
          │           │           │
          └─────┬─────┘           │
                ▼                 ▼
          对称加密密钥        身份证明
          (保密)            (认证)
```

## 三者关系矩阵

| 维度 | [[ECDH]] | [[ECDHE]] | [[ECDSA]] |
|------|----------|-----------|-----------|
| 解决什么 | 两方协商共享秘密 | 两方协商**一次性**共享秘密 | 证明"这消息是我发的" |
| 核心操作 | $d_A \cdot Q_B = d_B \cdot Q_A$ | 同 ECDH，但密钥对是临时的 | $s = k^{-1}(h + r \cdot d) \bmod n$ |
| 密钥生命周期 | 长期 | 单次会话 | 长期私钥 + 临时随机数 $k$ |
| [[前向保密]] | ✗ | ✓ | 不适用（签名不涉及会话密钥） |
| 关键随机数 | 无额外要求 | 临时密钥对（每次新生成） | $k$（绝不可重复，否则泄露私钥） |
| [[TLS]] 中的角色 | 已被 ECDHE 替代 | 密钥交换 | 身份认证 |
| 区块链中的角色 | 罕见 | 罕见 | 交易签名（secp256k1） |

## 互补关系

[[ECDHE]] 单独使用无法防御 [[中间人攻击]]——它保证"窃听者解不了"，但不保证"我在跟谁算"。[[ECDSA]] 恰好补上这个缺口：签名证明"这个临时公钥确实是某个持证方发出的"。

```
Alice                                    Bob
  │                                       │
  │──── ECDHE: 交换临时公钥 ────────────────│  ← 保密
  │                                       │
  │──── ECDSA: 用证书私钥签名握手摘要 ──────│  ← 认证
  │                                       │
  │════ AES-GCM: 用派生密钥加密数据 ═══════│  ← 安全通信
```

> [[ECDH]] · [[ECDHE]] · [[ECDSA]] · [[椭圆曲线离散对数问题]] · [[前向保密]] · [[中间人攻击]] · [[TLS]]

# 数字签名 — ECDSA

同一条椭圆曲线，[[ECDH]] 解决"协商秘密"，[[ECDSA]] 解决"证明身份"。数学地基一样（ECDLP），用法完全不同。

- 深入学习：[[数字签名-deeplearn]]
- 原子笔记：[[ECDSA]]

核心直觉：签名是"混合"（$k^{-1}$ 把消息哈希 $h$、私钥 $d$、随机数 $k$ 搅成一团），验证是"分离"（$s^{-1}$ 拆回消息路和身份路，看是否拼回原点）。

> [!warning] k 绝不能重复
> 两次签名用了同一个 $k$，联立方程直接解出私钥。PS3 破解事件的教训。

> [!note] 本节原子概念
> [[ECDSA]] · [[数字签名]] · [[Hash-then-Sign范式]] · [[secp256k1]]

| 6 | CA 可以为任意域名签证书 | **CT Log** / **DID+VC** | 事后审计 / 去中心化 | CT 有时间窗口；去中心化丢私钥没人救 |

# [新增] 涌现洞察

### 1. 保密与认证是同一个困难问题的两种消费方式

[[ECDH]] 和 [[ECDSA]] 共享 [[椭圆曲线离散对数问题|ECDLP]] 这个地基，但用法镜像对称：
- [[ECDH]]：双方各持私钥，交换公钥，算出**相同的**共享秘密 → 对称结构，双方地位平等
- [[ECDSA]]：一方用私钥签，所有人用公钥验 → 非对称结构，签名者 vs 验证者

一块地基，两栋房子。密钥交换消费的是"双方能独立算出同一个点"这个性质；签名消费的是"只有私钥持有者能产出特定点"这个性质。

### 2. 安全性经常不来自算法改进，而来自密钥生命周期策略

[[ECDH]] → [[ECDHE]] 的升级数学零变化，纯粹是把"复用密钥"改成"一次性密钥"。同样的模式出现在 DH → [[DHE]]。密码工程的瓶颈通常不在数学，在运维——密钥生成、存储、轮换、销毁的策略决定了实际安全水位。

### 3. 两处"临时随机数"，同形异质

[[ECDHE]] 的临时密钥对和 [[ECDSA]] 的随机数 $k$ 都叫"临时"、都要求"用完即毁"，但失败后果不对称：
- ECDHE 的临时密钥泄露 → **一次会话**暴露，其他会话不受影响
- ECDSA 的 $k$ 重复 → **长期私钥**永久暴露，所有历史签名不可信

同样的"新鲜随机数"要求，背后的威胁模型完全不同。这也解释了为什么 EdDSA 用确定性 nonce（$k = H(\text{private key}, \text{message})$）来消除 ECDSA 的这个薄弱点——对签名来说，确定性比随机性更安全。

### 4. ECDHE + ECDSA 是正交互补的最小完整对

缺 [[ECDHE]] → 没有 [[前向保密]]（旧版 RSA 密钥传输的问题）。缺 [[ECDSA]] → 无法防御 [[中间人攻击]]（裸 DH 的问题）。[[TLS]] 1.3 强制二者同时出现，正是因为保密和认证是两个独立的正交需求，砍掉任何一个都会在安全链上开洞。
