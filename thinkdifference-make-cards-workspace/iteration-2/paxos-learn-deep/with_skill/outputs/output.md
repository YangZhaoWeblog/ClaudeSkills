### [全景卡] Paxos 核心模块

**Q:** 画出 Paxos 协议的核心模块及消息流向。

**A:**
```
  Proposer
  (提议者)
      |
      | Prepare(n)
      v
  Acceptor x N          <- 需要多数派响应
  (接受者)
      |
      | Promise(n)
      v
  Proposer
      |
      | Accept(n, v)
      v
  Acceptor x N
      |
      | Accepted(n, v)
      v
  Learner
  (学习者)
```

### [场景卡] 为什么需要分布式共识协议？

**Q:** 在分布式系统中，为什么简单地"让所有节点投票决定"行不通？

**A:** 因为网络消息可能丢失或延迟，节点可能随时宕机。不同节点可能收到不同消息子集，导致各自基于不同信息做出决定，最终系统持有矛盾的状态。Paxos 正是为解决这个"不可靠网络下如何达成唯一一致决定"而设计。

---

### [what 卡] 多数派 (Quorum) 是什么？

**Q:** 在 Paxos 中，多数派 (Quorum) 机制是什么，它解决了什么问题？

**A:** 任何决定必须获得超过半数节点的同意（如 5 个节点中至少 3 个）。

关键性质：任意两个多数派之间至少有 1 个共同成员。

```
节点集合: {A, B, C, D, E}
多数派 1: {A, B, C}
多数派 2:    {B, C, D}
                ^---^ 必有交集
```

这保证了"知道旧决定的人"一定存在于新的多数派中，从而信息不会断链。

---

### [果卡] 多数派机制的数量边界

**Q:** 在 Paxos 中，5 个接受者的集群，多数派最少需要几个节点？3 个节点同意了值 A，另外 3 个节点能同意值 B 吗？

**A:** 最少需要 3 个节点。

```
5 个节点中:
多数派门槛 = floor(5/2) + 1 = 3

能否同时出现两个多数派？
  {A,B,C} 同意 A  -> 3个
  {C,D,E} 同意 B  -> 3个
  两组共用了 C，C 不可能同时承诺两件事
```

因此 3+3 不可能同时满足——任意两个多数派必然重叠，协议安全性由此保证。

### [what 卡] 提议编号 (Proposal Number) 的作用

**Q:** 在 Paxos 中，提议编号 (Proposal Number) 的规则是什么，它解决了什么竞争问题？

**A:** 每个提议必须携带全局唯一的递增编号。编号越大优先级越高。

作用：当多个提议者同时发起提议时，接受者只需要遵守"承诺只响应更高编号"的规则，就能自动淘汰旧提议，而不需要中央协调者。

---

### [why 卡] 提议编号为什么必须递增而非随机？

**Q:** 在 Paxos 中，提议编号为什么必须是递增的？如果两个提议者碰巧使用了相同编号会怎样？

**A:** 递增编号建立了"全序关系"——任意两个编号都能比出大小，接受者才知道该拒绝哪个、接受哪个。

若编号随机或相同：
- 接受者无法判断优先级
- 同编号的两个提议可能都被接受，导致不一致

实现上，提议者通常用 `(本地计数器, 节点ID)` 组合保证全局唯一且可排序。

### [what 卡] Paxos 两阶段流程

**Q:** 在 Paxos 协议中，从提议者发起提议到值被选定，完整的两阶段流程是什么？

**A:**
```
阶段一 (Prepare):
  Proposer ---Prepare(n)---> Acceptors
  Acceptor 检查: n > 已承诺的最大编号?
    是 -> 回复 Promise(n, 已接受的最高值或null)
         并承诺: 拒绝所有编号 < n 的请求
    否 -> 拒绝

阶段二 (Accept):
  收到多数派 Promise 后:
  Proposer ---Accept(n, v)---> Acceptors
    v = Promises 中携带的最高编号值（若有），否则 Proposer 自选
  Acceptor 检查: n >= 已承诺的最大编号?
    是 -> 回复 Accepted(n, v)，通知 Learner
    否 -> 拒绝
```

### [挖空卡] Paxos 端到端示例

**Q:** 3 个接受者 A、B、C，提议者 P1 提议"火锅"，C 一开始宕机，填写每一步发生了什么：

```
步骤 1: P1 发送 ________ 给 A, B, C
步骤 2: A, B 回复 ________，C 无响应 -> P1 获得多数派
步骤 3: P1 发送 ________ 给 A, B, C
步骤 4: A, B 接受；C 恢复后也接受 -> "火锅"______
```

**A:**
```
步骤 1: P1 发送 Prepare(1) 给 A, B, C
步骤 2: A, B 回复 Promise(1)，C 无响应 -> P1 获得多数派
步骤 3: P1 发送 Accept(1, "火锅") 给 A, B, C
步骤 4: A, B 接受；C 恢复后也接受 -> "火锅"被选定 (chosen)
```

### [反事实卡] 如果去掉 Prepare 阶段会怎样？

**Q:** 在 Paxos 中，如果去掉 Prepare 阶段、直接让提议者发送 Accept 消息，会出什么问题？

**A:** 两个提议者可能同时向不重叠的多数派发送不同的值：

```
P1 -> Accept(1, "火锅") -> {A, B, C}  ✓ 多数派
P2 -> Accept(2, "烤肉") -> {C, D, E}  ✓ 多数派
```

C 同时接受了两个不同的值，系统出现不一致。

Prepare 阶段的 Promise 机制的本质是：在正式提议前"锁定"接受者，使其承诺拒绝旧编号，从而保证同一时刻只有一个有效提议在推进。

---

### [果卡] Prepare 阶段中"携带已接受值"的作用

**Q:** 在 Paxos 的 Prepare 阶段，接受者在回复 Promise 时为什么要携带自己已接受的最高编号值？

**A:** 防止新提议者覆盖已接近被选定的值。

如果某个值 v 已被多数派接受（但新提议者不知道），新提议者必须继续提议 v 而不是自己的值。接受者通过在 Promise 中携带 `(已接受的最高编号, 对应的值)`，让新提议者发现并继承这个值，从而保证"已选定的值不会被推翻"。

### [边界卡] Paxos 的安全性 vs 活性

**Q:** 在 Paxos 中，如果 5 个接受者中有 3 个永久故障，协议会怎样？安全性和活性分别怎么保证？

**A:**
```
节点数: 5
故障数: 3
存活数: 2
多数派门槛: 3

结论: 2 < 3，无法组成多数派 -> 系统停止接受新提议
```

- **安全性 (Safety)**：已选定的值不会矛盾。Paxos 始终保证。即使停止工作，也不会输出错误结果。
- **活性 (Liveness)**：系统最终能达成决定。Paxos 不保证。节点故障或两个提议者持续互相抢占编号时，系统可能永远无法前进（活锁）。

### [连接卡] Paxos 与 Raft 的关系

**Q:** Raft 是什么？它和 Paxos 的核心区别是什么？

**A:** Raft 是 Paxos 的工程化简化，目标是"可理解性"。

```
Paxos:                    Raft:
任意节点可发起提议         只有 Leader 可发起提议
灵活但难以实现             受限但易于理解和实现
允许并发提议               串行化日志复制
```

Raft 用"限制谁能发起提议"换取了"日志顺序更清晰、选举逻辑更简单"，代价是在 Leader 选举期间系统不可用。

### [场景卡] 识别 Paxos 适用场景

**Q:** 哪类系统问题适合用 Paxos（或类似协议）来解决？给出两个真实系统的例子。

**A:** 凡是"多个节点需要就某个值/决定达成唯一共识"的场景：

- **分布式数据库的主节点选举**（如 Google Chubby、ZooKeeper）：多个节点竞争成为主节点，用 Paxos 保证只选出一个。
- **分布式日志复制**（如 etcd）：保证所有副本以相同顺序写入相同的日志条目。

判断标准：只要"脑裂（split-brain）"会造成数据损坏，就需要共识协议。

### [reverse 卡] 从 Promise 推断 Prepare 的目的

**Q:** 接受者在收到 Prepare(n) 后做出两个承诺：(1) 不再响应编号 < n 的 Prepare，(2) 不再接受编号 < n 的 Accept。从这两个承诺反推，Prepare 阶段的真正目的是什么？

**A:** Prepare 阶段的本质是"抢占式锁定"——

承诺 (1) 封锁旧提议者继续推进 Prepare。
承诺 (2) 封锁旧提议者继续推进 Accept。

两个承诺合在一起，相当于新提议者在发出正式提议前，先把旧提议者的"执行通道"全部关闭，确保后续的 Accept 阶段不会有旧值干扰。这就是为什么 Prepare 阶段是必要的：它是"写前加锁"的分布式等价物。

### [what 卡] Paxos 中学习者 (Learner) 的角色

**Q:** 在 Paxos 的三类角色中，学习者 (Learner) 是什么，它什么时候被通知，它自己参与投票吗？

**A:** 学习者是值被选定后的"接收方"，通常是实际执行决定的组件（如数据库状态机）。

- 不参与 Prepare/Accept 投票过程
- 当接受者收到 Accept 并形成多数派后，主动通知学习者
- 一个节点可以同时扮演多个角色（如既是接受者也是学习者）
