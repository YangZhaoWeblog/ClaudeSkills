# HTTP 缓存机制 闪卡集

---

## 面包屑地图

```
HTTP 缓存机制
├── 强缓存 (Strong Cache)
│   ├── Cache-Control（max-age）
│   ├── Expires
│   └── 优先级：Cache-Control > Expires
├── 协商缓存 (Conditional Cache)
│   ├── Last-Modified / If-Modified-Since（时间戳方式）
│   ├── ETag / If-None-Match（内容哈希方式）
│   └── 优先级：ETag > Last-Modified
├── 完整决策流程
│   └── 强缓存命中 -> 协商缓存验证 -> 重新请求
└── 实践策略
    ├── index.html：不缓存 / 短 max-age
    └── JS/CSS：内容哈希文件名 + 长期强缓存
```

## 核心卡片

### 全景卡

**Q：** 画出 HTTP 缓存机制的核心模块（顶层结构）。

**A：**
```
浏览器发起请求
      |
      v
 [强缓存检查]
  命中 -> 直接用本地缓存，不发请求
  未中 -> 发请求到服务器
      |
      v
 [协商缓存验证]
  未过期 -> 304，用本地缓存
  已过期 -> 200，返回新资源
```

---

### 卡片 01

**Q：** 在 HTTP 缓存中，强缓存（Strong Cache）的核心特征是什么？

**A：** 浏览器直接使用本地缓存，**不发任何请求到服务器**。

---

### 卡片 02

**Q：** 在 HTTP 强缓存中，控制缓存有效期的两个响应头是什么？

**A：** `Cache-Control`（如 `max-age=3600`）和 `Expires`。

---

### 卡片 03

**Q：** 在 HTTP 强缓存中，`Cache-Control` 和 `Expires` 同时存在时，浏览器遵循哪个？

**A：** `Cache-Control` 优先级更高，`Expires` 会被忽略。

---

### 卡片 04

**Q：** `Cache-Control: max-age=3600` 中的时间单位是什么，含义是什么？

**A：** 单位是**秒**。`max-age=3600` 表示该资源从响应时刻起，3600 秒（1 小时）内可直接使用本地缓存。

---

### 卡片 05

**Q：** 在 HTTP 缓存中，协商缓存（Conditional Cache）与强缓存的最关键区别是什么？

**A：** 协商缓存**会发请求到服务器**，让服务器验证本地缓存是否仍然有效；强缓存则完全不发请求。

---

### 卡片 06

**Q：** 在 HTTP 协商缓存中，基于"文件修改时间"的一对请求/响应头是什么？

**A：**
```
服务器响应头：Last-Modified（文件的最后修改时间）
浏览器请求头：If-Modified-Since（把上次的 Last-Modified 值带上）
```

---

### 卡片 07

**Q：** 在 HTTP 协商缓存中，基于"文件内容哈希"的一对请求/响应头是什么？

**A：**
```
服务器响应头：ETag（内容的哈希指纹）
浏览器请求头：If-None-Match（把上次的 ETag 值带上）
```

---

### 卡片 08

**Q：** 在 HTTP 协商缓存中，`ETag / If-None-Match` 和 `Last-Modified / If-Modified-Since` 同时存在时，服务器优先使用哪个？

**A：** `ETag / If-None-Match` 优先级更高。

---

### 卡片 09

**Q：** 在 HTTP 协商缓存中，`Last-Modified` 方案有什么已知的精度局限？

**A：** 精度只到**秒级**。如果文件在 1 秒内被修改多次，`Last-Modified` 无法区分，可能误判缓存有效。

---

### 卡片 10

**Q：** 在 HTTP 协商缓存验证中，服务器判断缓存**未过期**时，返回什么状态码？响应体有内容吗？

**A：** 返回 **304 Not Modified**，**不返回响应体**，浏览器直接使用本地缓存。

---

### 卡片 11

**Q：** 在 HTTP 缓存的完整决策流程中，强缓存未命中后，下一步是什么？

**A：** 发请求到服务器走**协商缓存验证**。若协商缓存也未命中（资源已更新），才重新获取完整资源（200）。

```
强缓存未命中
    |
    v
协商缓存验证
    |-----------> 304 -> 用本地缓存
    |-----------> 200 -> 用新资源
```

## 补充卡片

---

### 卡片 12

**Q：** 在 HTTP 缓存实践中，为什么 `index.html` 通常不做长期强缓存？

**A：** `index.html` 是入口文件，引用了带哈希的 JS/CSS 资源。如果 `index.html` 被长期强缓存，用户将无法获取新版本的入口，导致始终加载旧资源。

---

### 卡片 13

**Q：** 在前端部署实践中，JS/CSS 文件通常用什么策略实现"自动更新缓存"？

**A：** 用**内容哈希做文件名**（如 `app.abc123.js`），配合 `Cache-Control: max-age` 设为极长时间（如 1 年）。文件内容变化 -> 哈希变化 -> 文件名变化 -> 浏览器视为新资源，绕过强缓存。

---

### 卡片 14（挖空卡）

**Q：** 补全 HTTP 缓存优先级规则：

> 强缓存中：`______` 优先级高于 `Expires`
> 协商缓存中：`______` 优先级高于 `Last-Modified`

**A：**

> 强缓存中：`Cache-Control` 优先级高于 `Expires`
> 协商缓存中：`ETag` 优先级高于 `Last-Modified`

---

### 卡片 15

**Q：** 在 HTTP 缓存中，`ETag` 相比 `Last-Modified` 除了精度更高之外，还能处理哪类场景？

**A：** 文件内容**没有实质变化但修改时间更新**的场景（如文件被重新保存但内容未变）。`ETag` 基于内容哈希，内容不变则 ETag 不变，浏览器可继续使用缓存；而 `Last-Modified` 会误判为已更新。
