  面包屑地图：HTTP 缓存机制
================================================================

HTTP 缓存机制
|-- 强缓存（Strong Cache）
|   |-- 控制头：Cache-Control（如 max-age=3600）
|   |-- 控制头：Expires（绝对时间）
|   |-- 优先级：Cache-Control > Expires
|   +-- 行为：不发请求，直接使用本地缓存
|-- 协商缓存（Conditional Cache）
|   |-- 方式1：Last-Modified / If-Modified-Since（基于时间，精度到秒）
|   |-- 方式2：ETag / If-None-Match（基于内容哈希，精度更高）
|   |-- 优先级：ETag > Last-Modified
|   +-- 行为：发请求验证，未过期返回 304（无响应体）
|-- 完整决策流程
|   |-- 第一步：检查强缓存是否命中
|   |-- 第二步：未命中则走协商缓存
|   +-- 第三步：协商缓存未命中则重新请求资源
+-- 实践策略
    |-- index.html：不缓存或极短 max-age
    +-- JS/CSS：内容哈希文件名 + 长期强缓存

================================================================

[卡片1] what
Q: 在 HTTP 缓存机制中，强缓存（Strong Cache）和协商缓存（Conditional Cache）最根本的区别是什么？
A: 强缓存命中时，浏览器不发任何网络请求，直接读本地缓存。
   协商缓存命中时，浏览器必须发一次请求到服务器验证，服务器确认未过期才返回 304，
   浏览器再使用本地缓存。
   核心差异：有没有网络请求。


[卡片2] why
Q: 在强缓存中，为什么 Cache-Control 的优先级高于 Expires？
A: Expires 使用的是绝对时间（服务器时间），如果客户端和服务器时钟不一致，
   缓存行为会出错。
   Cache-Control 的 max-age 使用相对时间（从响应时刻起的秒数），
   不依赖时钟同步，更可靠，所以作为新标准优先级更高。


[卡片3] how（挖空）
Q: 在 HTTP 强缓存中，两个控制头是：Cache-Control（如 max-age=3600）和 ___。
   优先级关系：___ 高于 ___。
A: 另一个控制头是 Expires（绝对过期时间）。
   优先级：Cache-Control 高于 Expires。


[卡片4] what
Q: 在 HTTP 协商缓存中，ETag / If-None-Match 这对头的工作机制是什么？
A: 服务器在响应时，在 ETag 头中返回当前资源的内容哈希值。
   下次请求时，浏览器把这个哈希值放入 If-None-Match 头发给服务器。
   服务器对比哈希：一致则返回 304 Not Modified（无响应体），
   不一致则返回 200 和新资源。


[卡片5] what
Q: 在 HTTP 协商缓存中，Last-Modified / If-Modified-Since 这对头的工作机制是什么？
A: 服务器在响应时，在 Last-Modified 头中返回资源的最后修改时间。
   下次请求时，浏览器把这个时间放入 If-Modified-Since 头发给服务器。
   服务器对比时间：未修改则返回 304 Not Modified，
   已修改则返回 200 和新资源。


[卡片6] why
Q: 在协商缓存中，为什么 ETag 的优先级高于 Last-Modified？
A: Last-Modified 的精度只到秒。如果一个文件在 1 秒内被修改了多次，
   Last-Modified 无法感知，会错误地返回 304（认为未修改）。
   ETag 基于文件内容哈希，内容只要有一个字节变化，哈希就不同，精度更高。
   因此 ETag 更准确，优先级更高。


[卡片7] 果
Q: 在协商缓存中，如果只用 Last-Modified 而不用 ETag，在什么情况下会出现缓存错误？
A: 两种典型场景：
   1. 文件在 1 秒内被修改了多次——Last-Modified 相同，ETag 不同，
      此时应该更新但协商缓存错误地返回 304。
   2. 文件被重新生成但内容完全一致——Last-Modified 变了，ETag 相同，
      此时不需要更新但协商缓存错误地返回了新资源（浪费带宽）。


[卡片8] what
Q: 在 HTTP 协商缓存中，服务器返回 304 Not Modified 的含义是什么？
A: 304 表示"你本地缓存的版本还是最新的，直接用它"。
   服务器只返回响应头，不返回响应体（无资源内容），
   浏览器收到 304 后使用本地缓存的响应体。
   目的：节省带宽，只传输必要的头信息。


[卡片9] how（挖空）
Q: 在 HTTP 缓存的完整决策流程中：
   浏览器请求 → 检查强缓存 → 命中则直接用 → 未命中则走 ___ → 未命中则 ___
A: 未命中强缓存 → 走协商缓存（发请求验证）→ 未命中协商缓存 → 重新请求完整资源（200）


[卡片10] reverse
Q: 在 HTTP 缓存机制中，如果服务器想"立刻让用户的缓存失效、强制拉取新版本"，
   强缓存会带来什么障碍？怎么绕过？
A: 障碍：强缓存命中时浏览器根本不发请求，服务器无法主动通知客户端缓存失效。
   绕过方案：
   1. 给资源文件名加内容哈希（如 app.abc123.js），
      内容变化则文件名变化，新文件不在缓存中，浏览器会请求新资源。
   2. 通过 index.html 引入新的文件名（所以 index.html 本身不能强缓存）。


[卡片11] 场景
Q: 在前端部署策略中，为什么 index.html 通常设置为不缓存（或极短 max-age），
   而 app.abc123.js 这类文件却可以设置很长的强缓存？
A: index.html 是入口文件，引用了带哈希文件名的 JS/CSS。
   如果 index.html 被强缓存，用户就无法及时拿到新版本的 JS/CSS 文件名引用，
   导致仍然加载旧资源。
   而 app.abc123.js 的哈希名是内容的指纹：只要内容不变，文件名不变，
   可以永久缓存；内容一旦变化，文件名就变，旧缓存自然失效。
   本质：用 index.html 控制"更新通知"，用哈希文件名实现"安全的长期缓存"。


[卡片12] 果
Q: 在前端部署中，如果把 index.html 也设置了长期强缓存（如 max-age=31536000），
   会发生什么？
A: 用户在缓存过期前无法获取任何代码更新。
   即使部署了新版本（新的 JS/CSS 哈希文件），用户的浏览器仍然使用
   旧的 index.html，里面引用的还是旧的文件名，加载的还是旧代码。
   唯一解法是用户手动清除缓存，这在生产环境是不可接受的。

[卡片13] reverse
Q: 在 HTTP 协商缓存中，服务器返回 200 和服务器返回 304 的场景分别是什么？
   如何从响应反推"缓存是否被使用了"？
A: 返回 200：资源已修改（或首次请求），服务器返回完整响应体，本地缓存被替换。
   返回 304：资源未修改，服务器不返回响应体，浏览器使用本地缓存。
   反推规则：
   - 看到 304 → 浏览器正在使用本地缓存，网络只传了头
   - 看到 200（from cache）→ 强缓存命中，连请求都没发
   - 看到 200（正常）→ 缓存未命中，完整下载了新资源


[卡片14] 连接
Q: 在 HTTP 缓存机制中，强缓存和协商缓存分别对应 HTTP 头中的哪些字段？
A:
  强缓存（不发请求）：
    Cache-Control: max-age=N  （相对时间，优先）
    Expires: <绝对时间>        （绝对时间，次优先）

  协商缓存（发请求验证）：
    ETag / If-None-Match       （内容哈希，优先）
    Last-Modified / If-Modified-Since  （修改时间，次优先）

  注意：两对协商缓存头中，请求头和响应头成对出现：
    服务器响应用 ETag / Last-Modified 下发标识
    浏览器请求用 If-None-Match / If-Modified-Since 上传标识


[卡片15] what
Q: 在 HTTP 缓存中，协商缓存的两组头字段中，哪个是"服务器下发"的，哪个是"浏览器上传"的？
A: 服务器下发（响应头）：ETag、Last-Modified
   浏览器上传（请求头）：If-None-Match（对应 ETag）、If-Modified-Since（对应 Last-Modified）
   记忆规律：带 "If-" 前缀的是浏览器发出的"条件请求头"。
