# HTTP 缓存机制 - 闪卡

---

**Q1: HTTP 缓存分为哪两种机制？**

A: HTTP 缓存分为**强缓存**和**协商缓存**两种机制。

---

**Q2: 强缓存的核心特征是什么？**

A: 浏览器直接使用本地缓存，**不发请求到服务器**。

---

**Q3: 控制强缓存的 HTTP 头有哪两个？优先级如何？**

A: `Cache-Control`（如 `max-age=3600`）和 `Expires`。**Cache-Control 优先级高于 Expires**。

---

**Q4: 协商缓存的核心特征是什么？**

A: 浏览器**发请求到服务器**验证缓存是否过期，由服务器决定是否使用本地缓存。

---

**Q5: 协商缓存有哪两种方式？分别用什么 HTTP 头？**

A:
1. **Last-Modified / If-Modified-Since**：基于文件修改时间
2. **ETag / If-None-Match**：基于文件内容哈希

---

**Q6: Last-Modified 和 ETag 相比，哪个精度更高？为什么？**

A: **ETag 精度更高**。Last-Modified 精度只到秒；ETag 基于文件内容哈希，内容只要有变化就会不同。

---

**Q7: ETag 和 Last-Modified 哪个优先级更高？**

A: **ETag 优先级高于 Last-Modified**。

---

**Q8: 协商缓存验证未过期时，服务器返回什么？**

A: 服务器返回 **304 Not Modified**（响应体为空），浏览器使用本地缓存。

---

**Q9: 浏览器缓存的完整决策流程是什么？**

A:
1. 检查**强缓存** → 命中则直接用本地缓存
2. 未命中 → 走**协商缓存**（发请求验证）
3. 协商缓存也未命中 → **重新请求资源**

---

**Q10: 为什么 index.html 通常不设置长期缓存？**

A: 因为 index.html 是入口文件，如果长期缓存，用户无法及时获取到引用新版 JS/CSS 文件名的更新，导致页面加载旧资源。

---

**Q11: JS/CSS 文件如何实现既能长期缓存又能在更新时让浏览器获取新版本？**

A: 用**内容哈希做文件名**（如 `app.abc123.js`），配合长期强缓存（`Cache-Control: max-age` 设很大的值）。文件内容变了，哈希变了，文件名变了，浏览器就会请求新文件。

---

**Q12: Cache-Control: max-age=3600 表示什么？**

A: 资源在 **3600 秒（1小时）内**浏览器直接使用本地缓存，不向服务器发请求。

---

**Q13: 强缓存和协商缓存的本质区别是什么？**

A: **是否发送 HTTP 请求到服务器**。强缓存不发请求；协商缓存发请求但服务器可能返回 304 让浏览器用本地缓存。
