================================================================
  面包屑地图：HTTP 缓存机制
================================================================

HTTP 缓存机制
|-- 强缓存（Strong Cache）
|   |-- 触发条件：本地缓存未过期，直接使用，不发请求
|   |-- Cache-Control（如 max-age=3600）
|   +-- Expires（绝对时间戳）
|       +-- 优先级：Cache-Control > Expires
|-- 协商缓存（Negotiation Cache / Conditional Cache）
|   |-- 触发条件：强缓存未命中，发请求到服务器验证
|   |-- Last-Modified / If-Modified-Since
|   |   +-- 基于文件修改时间，精度到秒
|   |-- ETag / If-None-Match
|   |   +-- 基于文件内容哈希，精度更高
|   |-- 优先级：ETag > Last-Modified
|   +-- 验证结果：304 Not Modified（无响应体）或 200（返回新内容）
|-- 完整决策流程
|   |-- 1. 检查强缓存 -> 命中则直接使用
|   |-- 2. 强缓存未命中 -> 走协商缓存
|   +-- 3. 协商缓存未命中 -> 重新请求资源
+-- 工程实践
    |-- index.html：不缓存或极短 max-age
    +-- JS/CSS：内容哈希文件名（如 app.abc123.js）+ 长期强缓存

================================================================

[全景卡] 场景
Q: HTTP 缓存机制的整体结构是什么？
A:
HTTP 缓存机制
|-- 强缓存：不发请求，直接用本地缓存
|   +-- 控制头：Cache-Control > Expires（优先级）
|-- 协商缓存：发请求验证，命中返回 304
|   +-- 验证头：ETag/If-None-Match > Last-Modified/If-Modified-Since（优先级）
+-- 决策流程：强缓存 -> 协商缓存 -> 重新请求


[卡片1] what
Q: 强缓存和协商缓存最根本的区别是什么？
A: 是否发出 HTTP 请求。
   强缓存命中时，浏览器完全不发请求，直接读本地文件。
   协商缓存必须发一次请求到服务器确认缓存是否仍然有效。


[卡片2] why
Q: 为什么 HTTP 缓存要分两层（强缓存 + 协商缓存），而不只用一种？
A: 两者解决不同问题：
   强缓存追求极致性能——连网络请求都省了；
   协商缓存保证正确性——资源一旦更新，客户端能及时感知，同时节省传输带宽（304 不带响应体）。
   单用强缓存会有更新不及时的问题；单用协商缓存则每次都要发请求，浪费延迟。


[卡片3] how（挖空）
Q: 浏览器处理缓存的完整流程：
   发起请求 -> 检查强缓存 -> ___ -> ___ -> 使用响应
A: -> 强缓存命中则直接使用本地缓存，流程结束；
   -> 强缓存未命中则发请求走协商缓存；
   -> 服务器返回 304 则使用本地缓存，返回 200 则使用新内容。


[卡片4] what
Q: Cache-Control: max-age=3600 这个响应头的含义是什么？
A: 告诉浏览器：从收到响应起，3600 秒（1小时）以内，该资源被视为新鲜，直接使用本地缓存，不需要发任何请求。
   到期后强缓存失效，浏览器才会发请求验证。


[卡片5] why
Q: 为什么 Cache-Control 的优先级高于 Expires？
A: Expires 使用绝对时间（如 "Wed, 01 Jan 2025 00:00:00 GMT"），依赖客户端和服务器时钟一致。
   如果客户端系统时间偏差，缓存行为会出错。
   Cache-Control 的 max-age 使用相对时间（从收到响应算起的秒数），不受时钟偏差影响，更可靠。


[卡片6] why
Q: 为什么 ETag 的优先级高于 Last-Modified？
A: Last-Modified 精度只到秒——如果文件在 1 秒内被修改了两次，服务器无法区分。
   ETag 基于文件内容哈希，只要内容变了哈希就变，精度更高，也不依赖文件系统时间。
   但 ETag 会增加服务器计算成本，所以两者都支持时优先 ETag。


[卡片7] what
Q: 协商缓存验证成功时，服务器返回什么？浏览器做什么？
A: 服务器返回 304 Not Modified，响应体为空（节省带宽）。
   浏览器收到 304 后，使用本地已缓存的内容作为最终响应。


[卡片8] reverse
Q: 用户访问一个页面，完全没有发出任何网络请求就看到了内容。这是什么缓存机制在起作用？为什么？
A: 是强缓存命中。
   强缓存（Cache-Control 或 Expires）在有效期内，浏览器直接从本地读取资源，
   整个过程不产生任何 HTTP 请求——这正是强缓存和协商缓存的本质区别。


[卡片9] 果
Q: 如果 index.html 设置了很长的强缓存（如 max-age=31536000），会有什么后果？
A: 用户在缓存过期前永远看到旧版本的页面，即使服务器已经部署了新版本。
   因为 index.html 是入口文件，它引用的 JS/CSS 文件名也来自旧 HTML，
   导致新部署的代码无法被加载。这是缓存策略中最严重的错误之一。


[卡片10] 场景
Q: 一个前端项目发布新版本：index.html 和 app.abc123.js 应该分别用什么缓存策略？为什么？
A: index.html：不缓存（no-cache 或极短 max-age）。
   因为它是入口，必须每次都能获取到最新版本，才能加载到最新的 JS/CSS 文件名。

   app.abc123.js（内容哈希文件名）：长期强缓存（max-age=31536000）。
   文件名中包含内容哈希，内容一旦改变文件名就变，所以永远不会出现"旧缓存对应新内容"的问题。
   这是一种以"文件名变化"代替"缓存失效"的工程模式。

[卡片11] what
Q: 协商缓存中，If-Modified-Since 和 If-None-Match 是由谁发送、发给谁的？
A: 浏览器（客户端）在发送请求时携带这两个请求头，发给服务器。
   它们的值来自服务器上次响应时提供的 Last-Modified 和 ETag。
   这是一个"浏览器报出上次服务器给的凭证，服务器来验证"的过程。


[卡片12] 果
Q: 如果服务器不返回任何缓存相关的响应头（既无 Cache-Control 也无 ETag），浏览器怎么处理？
A: 行为由浏览器自行决定（启发式缓存，Heuristic Caching）。
   常见策略：取响应的 Date 和 Last-Modified 的差值的 10% 作为缓存时间。
   这是不确定行为，生产环境应显式设置缓存头，不依赖启发式策略。


[卡片13] 连接
Q: 304 Not Modified 和 200（from cache）在浏览器 DevTools 中都能见到，核心区别是什么？
A: 304 Not Modified：浏览器发出了请求，服务器验证后告知"没变，用你本地的"，有网络往返延迟。
   200 (from cache) / 200 (disk cache)：强缓存命中，浏览器完全没有发出请求，从本地读取。
   前者对应协商缓存，后者对应强缓存。
