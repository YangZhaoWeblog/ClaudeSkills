```
Paxos 共识协议
├── 1. 核心问题：不可靠网络中如何达成一致
├── 2. 三大构件
│   ├── 多数派机制（Quorum）
│   ├── 提议编号（Proposal Number）
│   └── 两阶段协议（Prepare-Accept）

```

## 核心卡片

---

### 全景卡

**Q:** 在分布式共识领域，Paxos 协议的顶层架构由哪几大模块组成？

**A:**

```
+----------+     +-----------+     +----------+     +----------+
|  角色    |     |  机制     |     |  协议    |     | 安全性   |
|          |     |           |     |          |     | 边界     |
| Proposer |     | Quorum    |     | Prepare  |     | 安全性 Y |
| Acceptor |     | 提议编号  |     | Accept   |     | 活性   N |
| Learner  |     | 承诺规则  |     | (两阶段) |     |          |
+----------+     +-----------+     +----------+     +----------+
```

四大模块：角色（谁参与）、机制（靠什么保证）、协议（怎么跑）、安全性边界（能保证什么/不能保证什么）。

---

### 场景卡（第一幕 -> 场景卡）

**Q:** Paxos 要解决的核心场景——5 个人通过短信远程投票决定聚餐地点，为什么"简单各自发消息"行不通？

**A:** 短信可能丢失、延迟，参与者可能随时掉线。没有协调机制时，不同的人可能收到不同的提议并各自以为达成了决定，导致最终结果不一致。

---

### 缺口卡（第二幕 -> 缺口卡 + 停顿点复用）

**Q:** 在 Paxos 的问题背景下，"简单多数投票"为什么无法保证一致性？

**A:** 如果不要求多数派重叠，可能出现两组不相交的人分别做出不同决定。比如 A、B 决定了火锅，C、D 决定了烤肉，两组决定互相矛盾。

---

**Q:** Paxos 的多数派机制（Quorum）中，5 个接受者至少需要几个人同意才能构成多数派？

**A:** 至少 3 个。因为 $\lceil 5/2 \rceil + 1 = 3$，任意两个 3 人组至少重叠 1 人，确保信息能在两个多数派之间传递。

---

**Q:** Paxos 的多数派机制中，如果 5 个接受者里已有 3 个同意了提议 A，另外 3 个人还能同意提议 B 吗？

**A:** 不能。3+3=6 > 5，两组必有重叠。重叠的那个人已经参与了 A 的多数派，在承诺机制下不会再接受编号更低的 B。这就是 Quorum 交集保证一致性的核心。

---

**Q:** Paxos 引入提议编号后，为什么编号必须是全局递增的，而不能是随机的？

**A:** 递增编号建立了全序关系——接受者可以明确判断"谁更新"。随机编号无法比较先后，接受者就无法决定该遵守哪个承诺、拒绝哪个提议。

---

**Q:** Paxos 的提议编号设计中，如果两个提议者碰巧使用了相同编号会怎样？

**A:** 协议无法区分两个提议，可能导致接受者对同一编号接受了不同的值。因此 Paxos 要求编号全局唯一，通常用 `(轮次, 提议者ID)` 组合来避免冲突。

---

**Q:** Paxos 为什么需要两阶段（Prepare + Accept），而不是一阶段直接发出提议？

**A:** 一阶段时，两个提议者可以同时向不同的多数派发送不同的值，导致两个多数派各自接受了矛盾的值。Prepare 阶段让接受者先做出承诺（"我不再接受编号更低的提议"），相当于先锁定通道，再传值，从而避免并发冲突。

---

### how 角度

**Q:** Paxos 协议的 Prepare 阶段，提议者具体发什么、接受者具体回什么？

**A:**
- 提议者发送 `Prepare(n)`：n 是提议编号，含义是"我想发起编号为 n 的提议，你同意吗？"
- 接受者回复 `Promise(n)`：含义是"我承诺不再接受编号 < n 的提议"。如果接受者之前已接受过某个值，还会附带返回该值。

---

**Q:** Paxos 协议的 Accept 阶段，提议者具体发什么、接受者具体怎么响应？

**A:**
- 提议者收到多数派的 Promise 后，发送 `Accept(n, v)`：n 是编号，v 是提议的值
- 接受者检查：如果没有对更高编号做过承诺，就接受并回复 `Accepted(n, v)`
- 当多数派都接受后，值 v 被选定（chosen）

---

**Q:** Paxos 的完整消息流中，学习者（Learner）的角色是什么？它在哪个阶段介入？

**A:** 学习者不参与投票决策。在 Accept 阶段结束后，接受者将 `Accepted(n, v)` 通知学习者。学习者收集到多数派的 Accepted 消息后，得知最终被选定的值。学习者是"只读"角色——只学习结果，不影响决策。

---

### 流程挖空卡（第三幕 -> 流程挖空卡）

**Q:** Paxos 端到端示例：提议者 P1 向 3 个接受者(A,B,C) 发送 Prepare(1)，A 和 B 回复了 Promise(1)，C 没收到。接下来 P1 应该做什么？为什么可以继续？

**A:** P1 已收到 2 个 Promise（A 和 B），达到 3 个接受者的多数派（2 >= $\lceil 3/2 \rceil + 1$... 实际 3 个节点多数派为 2），可以进入 Accept 阶段，发送 `Accept(1, "火锅")` 给 A、B、C。即使 C 没参与 Prepare，它仍可以在 Accept 阶段接受该提议。

---

**Q:** Paxos 中，接受者在 Prepare 阶段做出承诺后，收到一个编号更低的 Accept 请求时应如何处理？

**A:** 直接拒绝（忽略）。承诺的含义就是"我不再接受编号低于 n 的提议"。如果接受了，就会破坏一致性保证。

---

### what 角度

**Q:** 在 Paxos 协议中，"承诺"（Promise）的精确语义是什么？

**A:** 接受者对提议者的一个保证：从现在起，我不再响应任何编号小于 n 的 Prepare 请求，也不再接受任何编号小于 n 的 Accept 请求。这是 Paxos 安全性的关键锁。

---

**Q:** Paxos 协议中，一个值被"选定"（chosen）的精确条件是什么？

**A:** 一个值 v 被选定，当且仅当：某个提议 `(n, v)` 被多数派（超过半数）的接受者接受（Accepted）。一旦选定，不可更改。

---

### 果（安全性与活性）

**Q:** Paxos 协议能保证的安全性（Safety）具体指什么？

**A:** 两条保证：
1. **只有被提议过的值才能被选定**（非凭空产生）
2. **最多只有一个值被选定**（不会出现两个矛盾的决定）

无论网络如何延迟、丢包、节点如何崩溃恢复，这两条永远成立。

---

**Q:** Paxos 协议不能保证的活性（Liveness）具体指什么？在什么条件下活性会丧失？

**A:** 活性指"系统最终能达成决定"。Paxos 不保证活性——当多数派节点永久故障时，剩余节点无法组成 Quorum，系统将永远无法选定任何值。例如 5 个接受者中 3 个永久宕机，剩余 2 个 < 3（多数派门槛），协议阻塞。

---

### 边界卡（第四幕 -> 边界卡）

**Q:** Paxos 中，如果 5 个接受者中 2 个永久故障，协议还能正常工作吗？

**A:** 能。剩余 3 个节点仍可组成多数派（3 >= $\lceil 5/2 \rceil + 1 = 3$），协议正常运行。这是 Paxos 的容错上限：2f+1 个节点最多容忍 f 个故障，5 个节点最多容忍 2 个故障。

---

**Q:** Paxos 中，如果 5 个接受者中 3 个永久故障，系统处于什么状态？

**A:** 系统停止接受新提议（活性丧失），但已选定的值不会被推翻（安全性保持）。Paxos 的设计哲学：宁可停摆，不可出错。

---

### 反事实卡（第四幕 -> 反事实卡）

**Q:** Paxos 反事实推理——如果去掉 Prepare 阶段，只保留 Accept 阶段，会出什么问题？

**A:**
```
P1 --Accept(1,"火锅")--> A, B    (A,B 接受 -> 火锅多数派)
P2 --Accept(2,"烤肉")--> B, C    (B,C 接受 -> 烤肉多数派)
```
没有 Prepare 的承诺机制，B 可以同时接受两个不同的值，导致两个多数派各选了不同的值。Prepare 阶段正是为了在 Accept 之前建立排他性承诺，防止这种情况。

---

### reverse 角度卡

**Q:** Paxos 逆向推理——如果你观察到一个分布式系统中，某个已确认的决定在节点恢复后被推翻了，说明该系统违反了 Paxos 的哪个保证？根因最可能是什么？

**A:** 违反了安全性保证（最多只有一个值被选定）。根因最可能是：该系统没有正确实现承诺机制——节点恢复后没有持久化之前的承诺记录，导致它"遗忘"了旧承诺，接受了新的矛盾提议。Paxos 要求承诺必须持久化存储。

---

**Q:** 分布式系统设计中，如果你的需求是"任意时刻系统都不能产生矛盾结果，即使代价是暂时不可用"，这对应 Paxos 的哪个设计选择？

**A:** 对应 Paxos "安全性优先于活性"的设计哲学。Paxos 通过 Quorum + 承诺机制无条件保证安全性（不出矛盾），但允许在多数派不可用时牺牲活性（停止服务）。这也对应 CAP 定理中选择 CP（一致性 + 分区容忍）放弃 A（可用性）。

---

**Q:** 逆向推理——如果一个共识系统出现了两个不同的值都被"多数派接受"的情况，至少违反了 Paxos 的哪条规则？

**A:** 至少违反了承诺规则：某个接受者在对高编号做出 Promise 之后，仍然接受了低编号的提议。正是因为承诺保证了任意两个多数派的重叠节点会拒绝旧提议，才确保不会出现两个矛盾的多数派。

---

### 连接卡（第四幕 -> 连接卡）

**Q:** 在分布式共识领域，Paxos 和 Raft 的核心区别是什么？Raft 做了什么取舍？

**A:** Paxos 允许任何节点随时发起提议（去中心化），灵活但难理解、难实现。Raft 限定只有 Leader 能发起提议（中心化领导），牺牲了灵活性，换取了更清晰的理解模型和更简单的工程实现。本质上 Raft 是 Paxos 的一种工程化约束版本。

## 补充卡片

---

**Q:** Paxos 的提议编号通常如何生成以确保全局唯一？

**A:** 常用方案：`(轮次号, 提议者ID)` 的组合。轮次号本地递增，提议者 ID 全局唯一（如节点编号）。比较时先比轮次，轮次相同比 ID。这样无需全局协调就能生成唯一递增编号。

---

**Q:** Paxos 的 Prepare 阶段，如果接受者之前已经接受过某个提议 `(m, v_old)`，它在回复 Promise(n) 时还需要做什么？

**A:** 必须在 Promise 回复中附带之前已接受的提议 `(m, v_old)`。提议者收到后，必须采用所有回复中编号最大的那个已接受值作为自己的提议值，而不能用自己原本想提的值。这保证了已接近选定的值不会被覆盖。

---

**Q:** Paxos 中存在一种"活锁"（Livelock）场景——两个提议者交替用更高编号抢占，导致永远无法达成决定。这种情况如何解决？

**A:** 常见工程手段：选举一个"领导者"（Distinguished Proposer），同一时间只由一个提议者发起提议。这正是 Multi-Paxos 和 Raft 的做法——通过限制并发提议来避免活锁，代价是增加了领导者选举的复杂度。
