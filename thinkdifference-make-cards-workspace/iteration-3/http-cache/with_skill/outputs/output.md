## 面包屑地图

```
HTTP 缓存机制
├── 强缓存（Strong Cache）
│   ├── Cache-Control（max-age）
│   └── Expires
├── 协商缓存（Negotiation Cache）
│   ├── Last-Modified / If-Modified-Since
│   └── ETag / If-None-Match
├── 决策流程
│   └── 强缓存 -> 协商缓存 -> 重新请求
└── 实战策略
    ├── HTML：不缓存 / 短缓存
    └── 静态资源：内容哈希文件名 + 长期强缓存
```

## 核心卡片

### 卡 1 — 全景卡

**Q:** HTTP 缓存机制——整体由哪几大模块组成？

**A:**

```
+----------+    未命中    +----------+    未命中    +----------+
| 强缓存   | ----------> | 协商缓存  | ----------> | 重新请求  |
| (本地)   |             | (问服务器) |             | (全量)   |
+----------+             +----------+             +----------+
     |                        |
 Cache-Control           ETag / Last-Modified
 Expires                 304 Not Modified
```

三大模块：强缓存（本地判断）| 协商缓存（服务器验证）| 决策流程（串联二者）

---

### 卡 2 — what

**Q:** HTTP 缓存机制中——强缓存（Strong Cache）的核心行为是什么？

**A:** 浏览器直接使用本地缓存副本，**完全不发请求到服务器**。由响应头 Cache-Control 或 Expires 控制生效时间。

---

### 卡 3 — what

**Q:** HTTP 缓存机制中——协商缓存（Negotiation Cache）的核心行为是什么？

**A:** 浏览器**发请求到服务器**，携带缓存标识，由服务器判断缓存是否仍然有效。若有效返回 304（无响应体），浏览器使用本地缓存。

---

### 卡 4 — how

**Q:** HTTP 强缓存——通过哪两个响应头控制？优先级关系如何？

**A:**
- `Cache-Control`（如 `max-age=3600`，相对时间）
- `Expires`（绝对过期时间）

**Cache-Control 优先级高于 Expires**。两者同时存在时以 Cache-Control 为准。

---

### 卡 5 — how

**Q:** HTTP 协商缓存——两组验证字段分别是什么？各基于什么判断？

**A:**
| 验证方式 | 请求头 | 响应头 | 判断依据 |
|---|---|---|---|
| 方式一 | If-Modified-Since | Last-Modified | 文件修改时间（精度到秒） |
| 方式二 | If-None-Match | ETag | 文件内容哈希（精度更高） |

**ETag 优先级高于 Last-Modified**。

---

### 卡 6 — why

**Q:** HTTP 协商缓存中——为什么需要 ETag，Last-Modified 有什么不足？

**A:** Last-Modified 精度只到**秒级**。存在两个缺陷：
1. 1 秒内多次修改无法感知
2. 文件内容没变但修改时间变了（如重新保存），会导致不必要的资源重传

ETag 基于内容哈希，只要内容不变就不会失效。

---

### 卡 7 — how（流程）

**Q:** HTTP 缓存机制——浏览器请求资源时的完整决策流程是怎样的？

**A:**
```
浏览器发起请求
      |
 强缓存命中？--是--> 直接用本地缓存（200 from cache）
      |否
 发请求到服务器（携带 ETag/Last-Modified）
      |
 服务器验证通过？--是--> 返回 304，用本地缓存
      |否
 返回 200 + 全新资源
```

---

### 卡 8 — 果

**Q:** HTTP 协商缓存中——浏览器收到 304 Not Modified 响应，意味着什么？

**A:** 说明服务器验证了缓存标识（ETag 或 Last-Modified），确认资源**未发生变化**。服务器不返回响应体，浏览器直接使用本地缓存副本，节省带宽。

---

### 卡 9 — reverse

**Q:** HTTP 缓存排障——页面始终显示旧版本内容，最可能是哪个缓存头配置出了问题？

**A:** 最可能是 `Cache-Control: max-age` 设置过长，导致强缓存一直命中，浏览器根本不向服务器发请求验证。排查思路：
1. 检查该资源的 Cache-Control 响应头
2. 确认是否误给 HTML 页面设了长期缓存
3. 在 DevTools Network 面板看请求是否标记为 `(from disk cache)`

---

### 卡 10 — reverse

**Q:** HTTP 缓存机制——在 Network 面板看到 304 状态码，能反推出浏览器经历了哪些缓存阶段？

**A:** 可以反推出两件事：
1. **强缓存已失效**——否则不会发请求，状态码应是 200 (from cache)
2. **协商缓存命中**——服务器比对了 ETag 或 Last-Modified，确认资源未变

即：走完了「强缓存未命中 -> 协商缓存命中」这条路径。

---

### 卡 11 — 场景

**Q:** HTTP 缓存实战——为什么 index.html 和 JS/CSS 文件采用不同的缓存策略？

**A:** 因为它们的**更新发现机制不同**：
- `index.html` 是入口文件，浏览器必须先拿到它才知道要加载哪些资源。所以**不缓存或短缓存**，保证能及时发现更新。
- JS/CSS 文件名中嵌入了内容哈希（如 `app.abc123.js`），内容变了文件名就变了。所以可以放心用**长期强缓存**（如 `max-age=31536000`），因为新版本会通过新文件名自动绕开旧缓存。

## 补充卡片

---

### 补充卡 1 — what（优先级对比）

**Q:** HTTP 缓存机制中——存在两组优先级关系，分别是什么？

**A:**
- 强缓存内部：**Cache-Control > Expires**
- 协商缓存内部：**ETag > Last-Modified**

规律：较新引入的标准优先级更高（Cache-Control 和 ETag 都是 HTTP/1.1 引入）。

---

### 补充卡 2 — reverse（从需求反推方案）

**Q:** HTTP 缓存策略设计——如果要让静态资源既能长期缓存又能即时更新，应该怎么设计？

**A:** 采用**内容哈希文件名 + 长期强缓存**组合：
1. 构建时给文件名加内容哈希（如 `app.abc123.js`）
2. 设置 `Cache-Control: max-age=31536000`（一年）
3. 内容变化 -> 哈希变化 -> 文件名变化 -> 浏览器请求新 URL -> 自动绕开旧缓存

关键洞察：不是让缓存失效，而是让 URL 本身变化。
