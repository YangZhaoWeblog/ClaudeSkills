# 学透 PoW（工作量证明）

## 第一幕：问题感知

一群人在网上建了个公共记账本。没有银行、没有支付宝、没有任何中间人。每个人都可以往本子上写一笔"张三转给李四 10 块钱"。

问题来了：谁有权翻到下一页、把刚才这几笔交易正式记上去？

如果谁都能记，小明写"张三给我 100 块"，小红同时写"张三给我 100 块"——张三账上只有 100 块，但两页都被写上去了。更狠的，小明直接把昨天那页撕了，自己重写一页，把他付出去的钱又拿回来。

你大概会想到：投票。大家投票决定谁来记账。但这群人互相不认识，散布在全世界各地，随时可以换马甲，一个人可以注册一万个身份。投票？一个人投一万票。

那按"实名"来？谁来发这个"名"？一旦有人发名字，他就成了中间人——这群人建这个账本，开头就是为了摆脱中间人。

矛盾是：需要有人来记账，但不能指定谁来记；需要防止作弊，但没有人有资格当裁判。

生活中有没有这样一种场景：一群互不信任的人，需要公平选出一个人做某件事，而且这个选择过程本身不能依赖任何权威？

> **没有裁判，怎么选出记账人？**

## 第二幕：引导式构建

### 第一个工具：计算难题——用算力代替身份

回到那个困境：投票不行，因为身份可以伪造。需要一种"选举"方式，伪造不了，也拼不了人头。

什么东西伪造不了？**物理资源**。你可以注册一万个假名，但你不能凭空变出一万台电脑。你的算力就是你的"票"——不是按人头算，而是按你愿意烧掉多少电来算。

具体怎么做？出一道题。这道题有三个要求：

1. **做起来很费劲**——需要反复试错，没有捷径
2. **验证很容易**——别人一眼就能看出你做对了
3. **难度可调**——参与的人多了题变难，少了题变简单

这就是工作量证明（Proof of Work, PoW）的核心思想：不问你是谁，只看你做了多少"工"。谁先解出这道题，谁来记这一页账。

但什么题满足这三个条件？加法太简单。方程式有公式可以直接求解——没有试错，也就没有"工作量"。

我们需要一种运算：正着算极快，反着推只能猜。

> 想一想：你知道生活中哪些操作是"做容易、反推难"的？比如把鸡蛋打碎很容易，但看到一滩蛋液能还原出完整的鸡蛋吗？在数学或计算中，有没有类似的操作？

### 第二个工具：哈希函数——单向压缩机

正好有这样一种函数：哈希函数（Hash Function）。

把任意长度的数据丢进去，出来一个固定长度的"指纹"。比如 SHA-256，不管你输入一个字还是一整本书，输出永远是 256 位（64 个十六进制字符）。

关键性质：

- **单向性**：从输入算输出，一瞬间。从输出反推输入？没有任何已知的办法，只能一个一个猜。
- **雪崩效应**：输入改一个字母，输出面目全非。
- **确定性**：同样的输入，永远得到同样的输出。

举个例子（用 SHA-256 简化表示）：

```
输入: "Hello"       -> 输出: 185f8db32271fe25f...
输入: "Hello!"      -> 输出: 334d016f755cd6dc5...
输入: "Hello!!"     -> 输出: a]完全不同的一串...
```

只加了一个感叹号，输出就完全变了。你没法从输出里倒推任何关于输入的信息。

现在，如果我说："请你找到一个输入，使得它的哈希值小于某个数"——你除了不断尝试不同的输入、一遍遍算哈希，没有别的办法。

这就是那道"费力但可验证"的题。

> 想一想：如果有人宣称他找到了答案，你怎么验证？需要多长时间？和他找到答案花的时间比起来呢？

### 第三个工具：随机数（Nonce）——试错的手柄

现在我们有了哈希函数这个出题工具。但记账人要记的内容（交易记录）是固定的——这一页上就这些交易。如果输入固定，哈希值就固定，要么满足要求，要么不满足，没得"试"。

需要给记账人一个可以调节的旋钮：在账页内容后面加一个额外的数字，这个数字叫随机数（Nonce, Number used once）。记账人的工作就是不断更换这个 Nonce，反复计算：

```
Hash(交易记录 + Nonce_1) = 8a3f5c...  -> 不行
Hash(交易记录 + Nonce_2) = f291ab...  -> 不行
Hash(交易记录 + Nonce_3) = 00003d...  -> 前四位是零！
```

Nonce 是纯粹的试错变量。它不承载任何信息，唯一的作用就是让你有东西可以改。

> 想一想：如果不引入 Nonce，只用交易记录本身来算哈希，会出什么问题？记账人还能"工作"吗？

### 第四个工具：难度目标——可调的门槛

我们说"哈希值小于某个数"，这个"某个数"就是难度目标（Difficulty Target）。

哈希值可以看作一个很大的数。如果要求哈希值前面必须有 4 个零，那就是要求它小于 `0001000...000`。前面要求的零越多，满足条件的哈希值就越少，找到一个就越难。

用概率来理解：SHA-256 的输出有 $2^{256}$ 种可能，均匀分布。如果要求前 $k$ 位是零，满足条件的概率是 $\frac{1}{2^k}$。$k$ 越大，需要尝试的次数越多，"工作量"越大。

在比特币中，每 2016 个区块（大约两周）调整一次难度：如果这段时间内出块太快了（说明算力增加了），就把零的个数往上调；太慢了，就往下调。目标：平均每 10 分钟出一个块。

```
难度低:  Hash < 00000FFFFFFFFFFF...  (容易满足)
              |
              v
难度高:  Hash < 00000000000FFFFF...  (极难满足)
```

> 想一想：如果没有难度调节，所有人的电脑越来越快，会发生什么？出块速度会怎样变化？这对记账本有什么影响？

### 第五个工具：链式结构——让篡改代价指数级增长

现在有人能通过解题来争夺记账权了。但还有个问题：小明三天前记的那一页，他后悔了，想改掉。他重新算一个 Nonce 让那一页的哈希值也满足条件——岂不是就能篡改历史了？

解决办法：每一页的内容里，包含上一页的哈希值。

```
+----------------+      +----------------+      +----------------+
|   第 98 页     |      |   第 99 页     |      |   第 100 页    |
|                |      |                |      |                |
| 上页哈希: ...  | <--- | 上页哈希: H98 | <--- | 上页哈希: H99 |
| 交易记录       |      | 交易记录       |      | 交易记录       |
| Nonce: 8347   |      | Nonce: 52091  |      | Nonce: 11547  |
|                |      |                |      |                |
| 本页哈希: H98 |      | 本页哈希: H99 |      | 本页哈希: H100|
+----------------+      +----------------+      +----------------+
```

如果小明改了第 98 页的交易记录，那第 98 页的哈希值 H98 就变了（雪崩效应）。但第 99 页里记着"上页哈希: H98"——H98 变了，第 99 页也不合法了，得重新算。第 99 页改了，第 100 页也不合法了……

要改一页历史，就得重新算这一页之后所有页的哈希。每一页都需要巨大的算力。而在你重算的时候，全网其他人还在不断往后面加新页。你必须算得比全世界其他所有人加起来还快——除非你掌握全网 50% 以上的算力，否则你永远追不上。

这就是区块链（Blockchain）这个名字的由来——区块（Block）用链（Chain）串起来，越老的记录，被越多后续区块压着，篡改成本指数级增长。

> 想一想：如果有人真的掌握了全网 51% 的算力，他能做什么？他能凭空给自己创造钱吗？还是说他能做的事情比你想象的要有限？

### 第六个工具：最长链规则——分歧时的共识

最后一个缺口：如果两个记账人几乎同时解出了题，各自记了不同的一页，谁的算数？

规则很简单：**最长的那条链有效**。

两个人同时出了块，网络暂时分叉。一部分人收到了 A 的块先开始在 A 后面继续记账，另一部分人在 B 后面继续记账。但很快，其中一条链会先出下一个块——它变长了。所有人切换到更长的那条链，另一条被丢弃。

为什么"最长"就是"对的"？因为最长的链累积了最多的工作量。伪造一条更长的链意味着你需要比全网其他人加起来还多的算力，从你想篡改的那个区块开始重新算到超过当前最长链。

```
                    +------+     +------+
               +--->| B-99 |---->| B-100|  (较短，被丢弃)
               |    +------+     +------+
+------+       |
| #98  |-------+
+------+       |
               |    +------+     +------+     +------+
               +--->| A-99 |---->| A-100|---->| A-101|  (最长，胜出)
                    +------+     +------+     +------+
```

这也是为什么比特币交易通常建议等 6 个确认（后面又出了 6 个块）才算"真正确认"——6 个块之后，要推翻这笔交易需要追赶 6 个块的工作量，几乎不可能。

> 想一想：如果规则不是"最长链"而是"最先被我收到的链"，会有什么问题？攻击者可以怎样利用这个规则？

## 第三幕：完整组装

### 全景图

把第二幕引入的所有组件画在一起：

```
  [待记录的交易池]
        |
        v
+--------------------------------------------------+
|  候选区块                                         |
|                                                  |
|  +--------------------------------------------+  |
|  | 区块头                                      |  |
|  |                                            |  |
|  |  上一区块的哈希值 (链式结构)                  |  |
|  |  交易记录的摘要 (Merkle Root)                |  |
|  |  时间戳                                     |  |
|  |  难度目标                                   |  |
|  |  Nonce  <-- 不断递增尝试                     |  |
|  +--------------------------------------------+  |
|                                                  |
|  +--------------------------------------------+  |
|  | 交易列表                                     |  |
|  |  TX1: Alice -> Bob, 3 BTC                   |  |
|  |  TX2: Carol -> Dave, 1.5 BTC                |  |
|  |  ...                                        |  |
|  +--------------------------------------------+  |
+--------------------------------------------------+
        |
        | Hash(区块头) < 难度目标？
        |
   No --+-- Yes
   |         |
   v         v
 改 Nonce   广播区块 --> 其他节点验证 --> 接入最长链
 再算一次
```

六个组件的角色：

| 组件 | 角色 |
|------|------|
| 哈希函数 | 出题机制——单向性保证只能试错 |
| Nonce | 试错手柄——让矿工有东西可以改 |
| 难度目标 | 难度旋钮——控制出块速度 |
| 链式结构 | 篡改放大器——改一页要改后面所有页 |
| 最长链规则 | 分歧裁判——累积工作量最大的链胜出 |
| 工作量证明 | 以上五者组合的总称——用算力投票 |

### 端到端示例

用一个极简化的例子走一遍完整流程。假设哈希函数只输出 8 位十六进制数（实际 SHA-256 是 64 位），难度要求前 2 位为零。

**输入**：两笔待记录的交易

```
TX1: Alice -> Bob, 3 BTC
TX2: Carol -> Dave, 1.5 BTC
```

**步骤 1：组装候选区块**

```
上一区块哈希: 003a8f21
交易记录: [TX1, TX2]
Nonce: 0
```

**步骤 2：开始试错（挖矿）**

```
Nonce=0:  Hash("003a8f21|TX1,TX2|0")  = 7f3b2c19  -> 不满足(前2位非00)
Nonce=1:  Hash("003a8f21|TX1,TX2|1")  = e8a14d55  -> 不满足
Nonce=2:  Hash("003a8f21|TX1,TX2|2")  = 2c9e01af  -> 不满足
...
Nonce=47: Hash("003a8f21|TX1,TX2|47") = 005e83b1  -> 前2位是00！
```

矿工尝试了 48 次（Nonce 从 0 到 47），终于找到一个满足条件的哈希。

**步骤 3：广播区块**

矿工把这个区块（包含交易列表、Nonce=47、哈希值 005e83b1）广播到全网。

**步骤 4：其他节点验证**

任何节点收到后，只需算一次：

```
Hash("003a8f21|TX1,TX2|47") = 005e83b1  -> 前2位是00？ 是！ 合法。
```

验证只要 1 次计算，而找到这个答案平均需要 256 次（$16^2 = 256$ 种可能，期望尝试约 256 次）。做题难，验证易。

**步骤 5：接入区块链**

```
+----------------+      +------------------+
|  前一区块       |      |  新区块           |
|  哈希: 003a8f21| <--- |  上页哈希: 003a8f21|
|                |      |  TX1, TX2        |
|                |      |  Nonce: 47       |
|                |      |  哈希: 005e83b1  |
+----------------+      +------------------+
```

新区块的哈希 005e83b1 将成为下一个区块的"上页哈希"，链条继续延伸。

**步骤 6：难度调节**

如果最近出块太快（比如平均 5 分钟一个而不是 10 分钟），系统将难度从"前 2 位为零"调到"前 3 位为零"，满足条件的概率从 $\frac{1}{256}$ 降到 $\frac{1}{4096}$，出块变慢。

### 组装检查

- 哈希函数：第 2 步中反复计算 Hash()
- Nonce：第 2 步中从 0 递增到 47
- 难度目标："前 2 位为零"的要求
- 链式结构：第 5 步中新区块包含前一区块的哈希
- 最长链规则：隐含在第 4 步——节点验证后接在当前最长链后面
- 工作量证明：整个第 2 步的试错过程就是"证明你做了工作"

所有组件归位。

## 第四幕：验证与压力测试

### 反事实测试

**Q**: 如果去掉 Nonce，会怎样？

**A**: 没有 Nonce，区块内容固定，哈希值就固定。要么碰巧满足难度要求，要么永远不满足——矿工没有任何"旋钮"可以拧。挖矿变成了一个确定性事件而非概率试错过程。整个工作量证明机制就瘫痪了，因为没有"量"可以证明。

**Q**: 如果去掉链式结构（每个区块不再包含前一个区块的哈希），会怎样？

**A**: 每个区块变成了独立的页面。攻击者想修改历史中的某个区块，只需要重新算那一个区块的 Nonce——工作量和挖一个新块一样。篡改成本从"追赶整条链"降到"算一个块"，历史记录变得极其脆弱。链式结构是让"时间变成安全性"的关键——越老的记录越安全。

**Q**: 如果去掉难度调节，会怎样？

**A**: 如果难度固定不变，随着全球算力增长（更多矿工加入、硬件升级），出块速度越来越快。当出块间隔太短（比如每秒好几个块），网络中大量节点还没来得及收到上一个块就已经有新块了，分叉变成常态而非例外。链的共识崩溃，双花攻击变得容易。反过来，如果算力下降，出块可能慢到几小时甚至几天才出一个，系统几乎停摆。

### 边界测试

**Q**: 在什么条件下 PoW 会失效？

**A**: 核心失效条件有三个：

1. **51% 攻击**：当单一实体控制全网 50% 以上算力时，它可以持续制造比诚实矿工更长的链，实现双花——先用比特币买东西，等对方确认后发货，再用自己的算力构建一条不包含那笔交易的替代链、替换掉原链。注意：攻击者仍然不能凭空创造比特币（其他节点会验证交易合法性），也不能花别人的钱（没有私钥签名），他能做的主要是撤销自己的交易（双花）。

2. **哈希函数被攻破**：如果有人找到了一种不需要暴力枚举就能反推哈希输入的方法，"做题难"的前提消失了。当年 SHA-1 被证明存在碰撞弱点后，整个密码学界紧急迁移到 SHA-256。如果 SHA-256 也被攻破，PoW 的安全假设就不成立了。

3. **量子计算的威胁**：理论上，Grover 算法可以将哈希暴力搜索的复杂度从 $O(2^n)$ 降到 $O(2^{n/2})$。对于 SHA-256 来说，$2^{256}$ 降到 $2^{128}$——仍然天文数字，短期内不构成实际威胁。但这是一个值得持续关注的边界。

### 连接测试

**Q**: PoW 跟 PoS（权益证明, Proof of Stake）有什么关系？

**A**: 二者解决的是同一个问题——在无信任环境中选出记账人。区别在于"用什么资源投票"：

- PoW：用算力投票。你消耗的电力和硬件是你的"抵押品"。
- PoS：用持有的代币投票。你锁定的代币是你的"抵押品"。如果你作恶，代币被罚没。

PoW 的优点：安全假设简单（只需要诚实算力 > 50%），经过十几年实战检验。缺点：能源消耗巨大——比特币网络的年耗电量一度超过某些小国。以太坊在 2022 年从 PoW 切换到 PoS（The Merge），能源消耗降低了约 99.95%。

本质上，PoW 和 PoS 是"用什么不可伪造的稀缺资源来防止女巫攻击（Sybil Attack）"这个问题的两种不同回答。PoW 说"用物理世界的能量"，PoS 说"用系统内的经济质押"。

**Q**: PoW 跟 Hashcash 有什么关系？

**A**: Hashcash 是 Adam Back 在 1997 年发明的反垃圾邮件机制：发一封邮件前，你的电脑必须先做一道哈希难题。正常用户发几封邮件，几秒钟的计算无所谓；但垃圾邮件发送者每天要发百万封，每封都要算几秒，成本就扛不住了。这正是 PoW 的直接前身。中本聪在比特币白皮书中明确引用了 Hashcash，并把它从"反垃圾邮件"搬到了"分布式共识"的场景里。

### 终极挑战：空白重建

从第一幕的问题出发——"一群互不信任的人需要维护一个公共账本，没有中间人，谁来记账？"——你能否从空白重建整个 PoW 方案？
