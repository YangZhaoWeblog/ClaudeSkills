# 讲透快速排序

## 第一幕：问题感知

你手上有一沓 1000 张学生成绩单，需要按总分从高到低排好，发给教务处。

最笨的办法：从头到尾扫一遍，找出最高分的那张，抽出来放到一边；再扫一遍剩下的 999 张，找出第二高的；再扫 998 张……一千张成绩单，你得扫将近 50 万次。

你觉得太慢了。于是你换了个思路——每次拿相邻两张比，大的放前面、小的放后面，像冒泡一样一轮一轮地推。但算一算，还是差不多 50 万次比较。

问题出在哪里？每次操作，你只解决了一张纸的位置。一千张纸，你就得做将近一千轮。每一轮，你的眼睛几乎要扫过所有剩下的纸。

但你整理扑克牌的时候可不是这么干的。你会怎么做？大概是——随手抓一张，比它大的扔右边，比它小的扔左边。两堆分别再整理，很快就排好了。每次操作，你不是只解决一张纸的位置，而是把一整沓纸一分为二。

关键直觉就在这里：如果每一步都能把问题切成两半，1000 张纸只需要切大约 10 次（因为 $2^{10} = 1024$），每次切的时候扫一遍全部纸张——大约 1000 次 $\times$ 10 层 = 1 万次比较。从 50 万次降到 1 万次。

但问题也来了：那张"随手抓的牌"如果抓得不好呢？如果你抓到最小的一张，"比它小"的那堆是空的，"比它大"的那堆有 999 张——你根本没有把问题切成两半，等于白忙一场。

生活中有没有这样的操作：你在一堆东西中随手选一个"标杆"，然后所有东西都跟它比，比完以后自然分成了两拨，而这个标杆自己恰好落在了正确的位置上？

> **怎样一刀切下去，恰好把问题劈成两半？**

## 第二幕：引导式构建

### 第一个工具：分区（Partition）

第一幕给了我们一个直觉方向——"选一张牌，把其他牌分成两堆"。现在要把这个直觉变成可操作的步骤。

假设你面前有一排数字：

```
[8, 3, 7, 1, 5, 2, 6, 4]
```

我们先简单点：选最后一个数字 4 作为"标杆"。这个标杆有个名字，叫**基准元素（Pivot）**。

现在要做的事情：把所有比 4 小的数移到左边，把所有比 4 大的数移到右边，4 本身落在中间。做完以后，4 就待在了它最终应该待的位置——不管左边和右边内部怎么排，4 的位置不用再动了。

怎么做？用两根手指。一根叫 `i`，指向"下一个该放小数字的位置"，一开始指向最左边。另一根叫 `j`，从左往右扫描每一个数字。

规则很简单：`j` 每扫到一个比基准小的数，就把它和 `i` 指向的数交换，然后 `i` 往右挪一格。

走一遍：

```
基准 = 4, i = 0, j 从 0 开始扫描

[8, 3, 7, 1, 5, 2, 6, 4]
 j
 i

j=0: 8 >= 4, 不动
j=1: 3 < 4, 交换 arr[0] 和 arr[1] -> [3, 8, 7, 1, 5, 2, 6, 4], i=1
j=2: 7 >= 4, 不动
j=3: 1 < 4, 交换 arr[1] 和 arr[3] -> [3, 1, 7, 8, 5, 2, 6, 4], i=2
j=4: 5 >= 4, 不动
j=5: 2 < 4, 交换 arr[2] 和 arr[5] -> [3, 1, 2, 8, 5, 7, 6, 4], i=3
j=6: 6 >= 4, 不动

扫描结束，把基准 arr[7] 和 arr[i]=arr[3] 交换：
-> [3, 1, 2, 4, 5, 7, 6, 8]
              *
```

看看结果：4 落在了位置 3。它左边 [3, 1, 2] 全比它小，右边 [5, 7, 6, 8] 全比它大。**4 的最终位置已经确定了**——无论左右两边怎么排序，4 永远待在这里。

一次分区操作，扫描了 $n$ 个数字，做了 $n$ 次比较。代价是 $O(n)$。

> 想一想：分区结束后，左边那堆 [3, 1, 2] 内部是有序的吗？右边 [5, 7, 6, 8] 呢？分区操作本身解决了什么，又留下了什么没解决？

### 第二个工具：递归分治（Divide and Conquer）

分区操作搞定了一个数的最终位置，但左右两堆内部还是乱的。怎么办？

答案藏在问题本身里：左边那堆 [3, 1, 2] 不就是一个更小的"排序问题"吗？右边 [5, 7, 6, 8] 也是。而我们刚刚已经有了一个工具：选一个基准，分区。

对左边 [3, 1, 2] 再做一次分区——选 2 做基准，分完以后变成 [1, 2, 3]。对右边 [5, 7, 6, 8] 再做一次分区——选 8 做基准，分完以后变成 [5, 7, 6, 8]（8 到了最右边），然后对 [5, 7, 6] 再分区……

这就是**递归（Recursion）**：用同样的方法解决更小的同样问题。而整个策略叫做**分治（Divide and Conquer）**——把大问题切成小问题，各个击破。

用伪代码写出来：

```
quicksort(arr, lo, hi):
    如果 lo >= hi:       // 只剩 0 或 1 个元素，天然有序
        返回
    p = partition(arr, lo, hi)   // 分区，返回基准的最终位置
    quicksort(arr, lo, p - 1)    // 排左半边
    quicksort(arr, p + 1, hi)    // 排右半边
```

注意一个精巧之处：我们根本不需要"合并"步骤。归并排序（Merge Sort）是先切后合——切的时候不费力，合的时候费力。快速排序正好反过来——分区的时候做了所有工作（把小的扔左边、大的扔右边），分完就已经是有序的了，不需要合并。

> 想一想：归并排序是"先拆后合"，快速排序是"先分后递归"。如果你只有一块黑板（不能开辟额外空间），哪个更方便？为什么？

### 第三个工具：复杂度分析——为什么快？

现在我们有了完整算法。但它到底有多快？回到第一幕的直觉：关键在于"每次能不能把问题切成两半"。

**最好的情况**：每次基准恰好落在中间，数组被均匀切成两半。

```
                [n 个元素]                  -- 第 1 层：n 次比较
               /          \
        [n/2]              [n/2]            -- 第 2 层：n/2 + n/2 = n 次比较
       /     \            /     \
    [n/4]   [n/4]      [n/4]   [n/4]       -- 第 3 层：n 次比较
    ...                                     -- 共 log2(n) 层
```

每层做 $n$ 次比较，共 $\log_2 n$ 层，总共 $O(n \log n)$ 次比较。

**最坏的情况**：每次基准选到了最小（或最大）的那个，一边是空的，另一边是 $n-1$ 个元素。

```
[n 个元素]                      -- 第 1 层：n 次比较
 \
  [n-1 个元素]                  -- 第 2 层：n-1 次比较
   \
    [n-2 个元素]                -- 第 3 层：n-2 次比较
     \
      ...                       -- 共 n 层
```

总共 $n + (n-1) + (n-2) + \cdots + 1 = \frac{n(n-1)}{2} = O(n^2)$。退化成了冒泡排序的速度。

**平均情况**：数学证明，在随机输入下，快速排序的平均比较次数约为 $1.39 \, n \log_2 n$，仍然是 $O(n \log n)$。比最佳情况多了大约 39%，但仍然是同一量级。

> 想一想：如果输入数组已经是有序的（比如 [1, 2, 3, 4, 5, 6, 7, 8]），而你每次都选最后一个元素做基准，会发生什么？这是最好情况还是最坏情况？

### 第四个工具：基准选择策略——回应第一幕的核心矛盾

第三个工具暴露了一个关键问题：算法的命运系于基准的选择。选得好，$O(n \log n)$；选得差，$O(n^2)$。这正是第一幕收束问题的核心——"怎样一刀切下去，恰好劈成两半？"

几种策略：

**策略一：固定选第一个 / 最后一个**

最简单，但最危险。如果输入恰好是有序的（这在实际中非常常见——部分有序的数据到处都是），每次都选到极端值，直接退化为 $O(n^2)$。

**策略二：随机选取（Random Pivot）**

从当前区间里随机挑一个元素做基准。这个策略的精妙之处在于：它不保证每一次都切得好，但它保证**没有任何特定输入**能让你稳定地切得差。最坏情况仍然存在（你可能运气极差），但概率极低——就像连续抛硬币 1000 次全部正面一样不太现实。

用概率的语言说：随机化快速排序的**期望**时间复杂度是 $O(n \log n)$，这个期望是对算法自身的随机性求的，与输入无关。

**策略三：三数取中（Median of Three）**

取第一个、中间那个、最后一个，三个数的中位数做基准。这几乎不增加任何开销，但有效避免了有序输入的最坏情况。实际工程中最常用。

**策略四：Ninther（三组三数取中的中位数）**

取 9 个元素，分三组各取中位数，再取这三个中位数的中位数。在超大数组上用，成本稍高但分区更均匀。

> 想一想：假设有人恶意构造输入数据，专门让你的快速排序跑出 $O(n^2)$。在策略一和策略二下，他分别能不能得逞？为什么？

### 第五个工具：原地排序与空间复杂度

到目前为止，我们的分区操作是在原数组上完成的——只交换元素，不需要额外数组。这叫**原地排序（In-place Sort）**。

但递归本身是有代价的。每一层递归调用会在**调用栈（Call Stack）**上占一个位置，保存当前的 `lo`、`hi` 等信息。

最好情况：递归深度 $O(\log n)$，栈空间 $O(\log n)$。

最坏情况：递归深度 $O(n)$，栈空间 $O(n)$——如果数组有一百万个元素，栈可能炸掉（栈溢出，Stack Overflow）。

解决办法：**尾递归优化（Tail Recursion Optimization）**——每次递归时，先处理较短的那一半（用递归），再处理较长的那一半（用循环替代递归）。这样递归深度永远不超过 $O(\log n)$。

```
quicksort(arr, lo, hi):
    while lo < hi:
        p = partition(arr, lo, hi)
        if p - lo < hi - p:        // 左半边更短
            quicksort(arr, lo, p - 1)  // 递归处理短的
            lo = p + 1                 // 循环处理长的
        else:                       // 右半边更短
            quicksort(arr, p + 1, hi)
            hi = p - 1
```

> 想一想：为什么总是递归处理"短的那一半"就能保证栈深度不超过 $O(\log n)$？提示：每次递归处理的那一半，最多有多大？

## 第三幕：完整组装

### 全景图

```
                    quicksort(arr, lo, hi)
                            |
                   +--------+--------+
                   |                 |
              lo >= hi ?          lo < hi
              (返回)                 |
                            选基准 (Pivot)
                       [固定/随机/三数取中]
                                |
                         partition(arr, lo, hi)
                   用双指针把数组分成三段：
                   [< pivot | pivot | > pivot]
                                |
                   +------------+------------+
                   |                         |
          quicksort(左半边)          quicksort(右半边)
          arr[lo .. p-1]            arr[p+1 .. hi]
```

**组件清单**：

| 组件 | 作用 | 代价 |
|------|------|------|
| 基准选择 | 决定"从哪里切" | $O(1)$ 到 $O(1)$（取决于策略） |
| 分区操作 | 一趟扫描，把数组一分为三 | $O(n)$（线性扫描） |
| 递归分治 | 对左右两半重复上述过程 | 递归 $O(\log n)$ 层 |
| 尾递归优化 | 保证栈深度不爆 | 空间 $O(\log n)$ |

**总复杂度**：

|  | 时间 | 空间 |
|--|------|------|
| 最好 | $O(n \log n)$ | $O(\log n)$ |
| 平均 | $O(n \log n)$ | $O(\log n)$ |
| 最坏 | $O(n^2)$ | $O(n)$（优化后 $O(\log n)$） |

### 端到端示例

**输入**：`[5, 3, 8, 1, 2, 7, 4, 6]`，使用"选最后一个元素做基准"的策略。

**第一轮分区**：基准 = 6

```
[5, 3, 8, 1, 2, 7, 4, 6]

i=0, j 从 0 开始：
j=0: 5 < 6, swap(arr[0], arr[0]), i=1  -> [5, 3, 8, 1, 2, 7, 4, 6]
j=1: 3 < 6, swap(arr[1], arr[1]), i=2  -> [5, 3, 8, 1, 2, 7, 4, 6]
j=2: 8 >= 6, 不动
j=3: 1 < 6, swap(arr[2], arr[3])  i=3  -> [5, 3, 1, 8, 2, 7, 4, 6]
j=4: 2 < 6, swap(arr[3], arr[4]), i=4  -> [5, 3, 1, 2, 8, 7, 4, 6]
j=5: 7 >= 6, 不动
j=6: 4 < 6, swap(arr[4], arr[6]), i=5  -> [5, 3, 1, 2, 4, 7, 8, 6]

把基准 6 放到位置 i=5：swap(arr[5], arr[7])
-> [5, 3, 1, 2, 4, 6, 8, 7]
                     *
```

6 在位置 5，已归位。左半边 [5, 3, 1, 2, 4]，右半边 [8, 7]。

**第二轮（左半边）**：`[5, 3, 1, 2, 4]`，基准 = 4

```
i=0:
j=0: 5 >= 4, 不动
j=1: 3 < 4, swap(arr[0], arr[1]), i=1 -> [3, 5, 1, 2, 4]
j=2: 1 < 4, swap(arr[1], arr[2]), i=2 -> [3, 1, 5, 2, 4]
j=3: 2 < 4, swap(arr[2], arr[3]), i=3 -> [3, 1, 2, 5, 4]

基准归位：swap(arr[3], arr[4])
-> [3, 1, 2, 4, 5]
               *
```

4 归位。左半边 [3, 1, 2]，右半边 [5]（单元素，已有序）。

**第三轮（左半边的左半边）**：`[3, 1, 2]`，基准 = 2

```
i=0:
j=0: 3 >= 2, 不动
j=1: 1 < 2, swap(arr[0], arr[1]), i=1 -> [1, 3, 2]

基准归位：swap(arr[1], arr[2])
-> [1, 2, 3]
       *
```

2 归位。左半边 [1]，右半边 [3]，都是单元素。

**第二轮（右半边）**：`[8, 7]`，基准 = 7

```
i=0:
j=0: 8 >= 7, 不动

基准归位：swap(arr[0], arr[1])
-> [7, 8]
    *
```

**最终结果**：`[1, 2, 3, 4, 5, 6, 7, 8]`

### 组装检查

- 基准选择：每一轮都用到了（选最后一个元素）
- 分区操作：每一轮的核心操作，双指针扫描 + 交换
- 递归分治：对左右两半分别调用 quicksort
- 尾递归优化：本例规模小未触发，但在大规模数据上保证栈安全
- 复杂度：本例共 8 个元素，分了约 4 层（实际因分区不完美略多），总比较次数约 20 次，远小于冒泡排序的 $\frac{8 \times 7}{2} = 28$ 次

## 第四幕：验证与压力测试

### 反事实测试

**Q**: 如果去掉分区操作，只保留递归框架，会怎样？

**A**: 没有分区，递归就不知道该把什么放左边、什么放右边。递归框架只是"拆问题"的骨架，分区才是"做事"的肌肉。没有分区的快速排序，就像只有目录没有正文的书——结构在，内容没有。你会无限递归但永远不会把任何一个元素放到正确位置。

**Q**: 如果分区操作不把基准放到最终位置（只分成"小的一堆"和"大的一堆"，基准随便扔），会怎样？

**A**: 这个基准在后续递归中会被反复处理，永远无法确定自己的位置。更严重的是，递归可能不收敛——如果基准每次都被扔进某一堆，问题规模不缩小，算法可能死循环或栈溢出。基准归位是递归收敛的保证：每一轮至少有一个元素到达最终位置，问题规模严格缩小。

### 边界测试

**Q**: 在什么条件下快速排序会退化为 $O(n^2)$？

**A**: 核心条件是**分区极度不均匀**。具体场景包括：

1. **输入已有序 + 固定选首/尾做基准**：每次基准都是最小或最大值，一边为空，另一边 $n-1$ 个元素。经典的最坏情况。
2. **所有元素相同**：基准等于所有元素，标准分区把所有元素都扔到一边，同样退化。解决办法是**三路分区（Three-way Partition）**，把数组分成"小于、等于、大于"三段，等于基准的元素直接跳过。
3. **恶意构造的输入**：如果攻击者知道你的基准选择策略，可以构造输入让你每次都选到最差基准。随机化选取是对抗这种攻击的标准手段。

**Q**: 快速排序是稳定排序吗？

**A**: 不是。**稳定排序（Stable Sort）**要求相等元素保持原始相对顺序。在分区过程中，交换操作会打乱相等元素的相对位置。比如 [3a, 3b, 1]（3a 和 3b 值相同但原始顺序 a 在 b 前面），分区后可能变成 [1, 3b, 3a]。如果需要稳定性，用归并排序。

### 连接测试

**Q**: 快速排序跟归并排序有什么关系？

**A**: 二者都是分治策略，但分治的"发力点"不同：

| | 快速排序 | 归并排序 |
|--|---------|---------|
| 拆分 | 费力（分区） | 不费力（从中间切） |
| 合并 | 不费力（不需要） | 费力（归并两个有序数组） |
| 稳定性 | 不稳定 | 稳定 |
| 空间 | $O(\log n)$（原地） | $O(n)$（需要额外数组） |
| 最坏时间 | $O(n^2)$ | $O(n \log n)$ |
| 实际速度 | 通常更快（缓存友好） | 常数因子较大 |

快速排序在实践中通常更快，原因是它对数组进行原地操作，**缓存局部性（Cache Locality）**好——CPU 缓存更喜欢连续访问同一块内存。归并排序需要在原数组和辅助数组之间来回复制，缓存命中率低。

**Q**: 快速排序跟二叉搜索树（BST）有什么关系？

**A**: 快速排序的递归过程，本质上就是在构建一棵二叉搜索树。每次选的基准就是树的根节点，左半边是左子树，右半边是右子树。快速排序的最好情况对应平衡 BST，最坏情况对应退化成链表的 BST。随机化快速排序等价于随机插入构建 BST。

### 终极挑战：空白重建

从第一幕的问题出发——"排序太慢，每次操作只搞定一个元素的位置"——你能否从空白重建整个方案？

**重建路线图**：

```
[排序太慢，O(n^2)]
       |
       v
[直觉：能不能每次把问题切成两半？]
       |
       v
[工具1：分区——选一个基准，小的扔左边，大的扔右边]
   "但左右两半内部还是乱的"
       |
       v
[工具2：递归——对左右两半重复同样的操作]
   "但到底有多快？"
```
